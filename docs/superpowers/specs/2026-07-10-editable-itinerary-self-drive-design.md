# AeroTravel 可编辑行程与自驾路线设计

日期：2026-07-10

状态：已批准（2026-07-10）

## 1. 背景

AeroTravel 当前擅长一次性生成旅行规划，但生成结果基本只读。用户不能自然地加入自己想去的景点，也不能在保持已有安排的前提下局部调整。重新生成会整体替换行程，导致用户只修改一个地点时，其他日期、景点和顺序也可能发生变化。

现有自驾能力只存在于城际交通选项中，还不是一个完整的自驾规划工作流。用户需要输入若干城市和景点，让系统给出合理的道路顺序、按天拆分、真实道路距离、预计驾驶时间和过路费，同时仍能手动排列节点。

这两个问题具有同一个根因：产品缺少结构化、可编辑、带约束的行程草稿。解决方案不是继续增强一次性 AI 提示词，而是把 AeroTravel 从“行程生成器”升级为“AI 生成初稿、用户持续编辑、系统按约束局部优化”的规划工作台。

## 2. 目标

- AI 生成后，用户可以搜索、粘贴名称或在地图上选点，将地点加入想去清单。
- 用户可以将地点安排到某一天、跨天移动，并调整当天顺序。
- 用户可以分别设置必去、固定日期、固定时段和固定顺序。
- 用户修改后，系统不自动重排行程，而是提供“仅保存”和“智能优化”两个明确动作。
- 智能优化只能调整所选范围内未锁定的节点，并在应用前展示差异。
- 自驾模式支持环线和单程，以及少开车、均衡、多体验三种策略。
- 自驾模式展示真实道路距离、预计驾驶时间、过路费和按天路线分段。
- 用户手动排列自驾节点后，无需调用 AI 即可重新计算道路指标。
- 所有编辑继续保存在浏览器 `localStorage`，不引入数据库、账号系统或必需的新付费服务。
- 保留现有 `/api/plan`、`applyPlan()`、单一可变 `state` 和无构建前端模式。

## 3. 非目标

- 首版不提供逐向导航、实时路况导航或车辆控制。
- 首版不承诺直接跳转高德导航；只预留稳定的有序途经点和路线几何数据边界。
- 不提供租车、酒店、门票或停车位预订。
- 不增加多人协作、跨设备同步、数据库或登录系统。
- 不追求任意数量节点的数学全局最优解。产品面向常见的小型旅行路线，使用可解释且可重复的约束启发式算法。
- 不允许 AI 直接决定道路最短路径，也不允许 AI 绕过用户硬约束。
- 首版单条自驾路线最多包含 20 个已定位节点。超过上限时要求用户拆分路线，避免不可控的路网请求量和等待时间。

## 4. 核心产品决策

### 4.1 修改后的行为

用户的增删、拖动和约束修改先进入编辑草稿，不立即改变已应用行程。底部固定操作区提供：

- `撤销`：撤销最近一次草稿操作。
- `仅保存`：完全按当前手工编辑结果应用，不进行自动重排。
- `智能优化`：基于当前草稿生成候选方案，展示变化后由用户决定是否应用。

系统不得在一次普通拖动后自动请求 AI 或整体替换行程。

仅保存仍会运行本地结构校验和现有质量检查。未知节点、重复节点等结构错误必须修复后才能应用；时间重叠、路线过满等可解释冲突以警告展示，用户确认后仍可按自己的顺序保存，系统不得借校验之名自动重排。

### 4.2 普通行程布局

沿用现有“左侧行程、右侧地图”的桌面布局。结果区增加浏览和编辑模式：

- 编辑模式左侧为可收起的“想去清单”。
- 编辑模式右侧为当前日期的有序日程。
- 用户可以把想去地点拖入某一天，也可以通过移动菜单完成相同操作。
- 地图同时显示已安排节点与未安排节点，二者使用不同视觉状态。
- 移动端使用日期切换、移动菜单和上移/下移操作，不依赖精确拖拽。

### 4.3 自驾布局

顶部提供“城市行程 / 自驾路线”模式切换。自驾模式复用相同工作台：

- 左侧按顺序展示城市和景点节点。
- 节点可以拖动、上移、下移、编辑、锁定或删除。
- 环线固定同一起终点；单程分别指定起点和终点。
- 路线列表显示按天分段位置。
- 右侧地图显示真实道路路线以及总里程、总驾驶时间和总过路费。
- 策略为少开车、均衡、多体验，默认均衡。

### 4.4 地点添加

提供三种入口：

- 调用现有高德 POI 搜索。
- 在地图上点选位置，并通过后端反向地理编码补充名称和城市。
- 粘贴或输入地点名称，再尝试搜索解析。

无法获得可信坐标的地点保留在想去清单中并标记为“待定位”。它可以保存和编辑，但在定位前不参与路线优化。

用户通过搜索、地图点选或名称输入主动加入的地点默认设置为必去，用户可以显式改为可选。AI 初稿和后续 AI 推荐的地点默认可选，只有用户标记后才成为必去。这样优化器不会删除用户刚刚主动加入的地点。

## 5. 状态模型

### 5.1 三段式状态

浏览器维护三个彼此分离的状态：

1. `AppliedPlan`：当前已经确认，可用于浏览、复制、导出和保存快照的行程。
2. `WorkingDraft`：用户正在编辑的草稿。地图可以预览它，但复制和导出仍以 `AppliedPlan` 为准。
3. `CandidatePlan`：优化接口返回的候选草稿。只有用户确认后才会替换 `AppliedPlan` 和 `WorkingDraft`。

现有 `state.itinerary` 继续承担 `AppliedPlan` 的角色。`state` 增加草稿、候选、编辑历史和编辑模式字段。`applyPlan()` 仍是 AI 生成结果和本地快照的主 hydration 入口；它完成现有映射后，再通过 `itineraryToDraft()` 纯函数创建对应草稿。用户选择仅保存或应用候选方案时，通过 `draftToItinerary()` 生成兼容现有渲染、复制和导出的行程，再更新 `state.itinerary`。

每次草稿操作增加单调递增的 `revision`。优化请求携带基础 revision；响应 revision 与当前草稿不一致时必须丢弃，防止较慢的旧请求覆盖新编辑。

### 5.2 `TripDraft`

以下字段名是 Pydantic 和 JavaScript 共用的规范名称：

```json
{
  "schema_version": 2,
  "id": "trip-id",
  "revision": 12,
  "mode": "itinerary",
  "route_shape": "one_way",
  "strategy": "balanced",
  "start_date": "2026-07-18",
  "city_stops": [],
  "nodes": [],
  "days": [],
  "route": null
}
```

`mode` 为 `itinerary` 或 `self_drive`。`route_shape` 为 `one_way` 或 `round_trip`。`strategy` 为 `efficient`、`balanced` 或 `experience`。

### 5.3 `PlaceNode`

每个地点使用稳定 ID，不能使用数组下标或名称作为身份标识：

```json
{
  "id": "node-id",
  "source": "amap_search",
  "provider_id": null,
  "name": "西溪湿地",
  "city_id": "city-hangzhou",
  "city": "杭州",
  "location": {
    "lat": 30.2668,
    "lng": 120.0624,
    "status": "resolved"
  },
  "status": "wishlist",
  "duration_minutes": 180,
  "schedule": {
    "day_id": null,
    "time_window": null
  },
  "constraints": {
    "required": true,
    "fixed_day": false,
    "fixed_time": false,
    "fixed_order": false
  },
  "manual_rank": null,
  "metadata": {}
}
```

`source` 区分 AI、高德搜索、地图点选和手工输入。`status` 为 `wishlist`、`scheduled` 或 `removed`。普通拖动只更新顺序和 `manual_rank`，表达软偏好；只有显式开启 `fixed_order` 才形成硬约束。

所有外部 metadata 继续经过现有清理和转义路径，不把高德或 AI 文本直接插入 HTML。

### 5.4 `CityStop`

`TripDraft.city_stops` 是现有 `state.cities` 的草稿等价物，在编辑期间保存待应用的城市顺序和每城天数：

```json
{
  "id": "city-hangzhou",
  "name": "杭州",
  "days": 3,
  "transport": "driving",
  "fixed_order": false
}
```

用户可以手动调整城市顺序。按日或按城市优化不得改变 `city_stops` 顺序；只有全行程优化可以建议调整未锁定城市，并且仍需经过候选差异确认。应用草稿后，`city_stops` 必须同步回现有 `state.cities`；对于已应用行程，`state.cities[].days` 继续是每城天数的运行时来源，全局总天数仍由各城市天数相加得到。

### 5.5 `DayDraft`

每天保存节点 ID 的有序列表，而不是复制完整地点对象：

```json
{
  "id": "day-2",
  "day": 2,
  "date": "2026-07-19",
  "primary_city_id": "city-hangzhou",
  "node_ids": ["node-a", "node-b"],
  "max_driving_minutes": 240
}
```

`node_ids` 是当天顺序的唯一来源。自驾按天拆分时，`max_driving_minutes` 用于策略约束；普通行程可以为空。

地点的事实城市由 `PlaceNode.city_id` 决定，拖到其他城市的日期不会改写它。普通行程中，跨事实城市拖动需要用户确认并把该日标记为跨城日；自驾模式允许自然形成多城市日。优化器必须据此加入城际移动成本，不能把地点静默归入错误城市。

### 5.6 道路结果

道路结果必须区分真实数据与估算数据：

```json
{
  "source": "amap",
  "status": "provider",
  "ordered_node_ids": ["node-a", "node-b"],
  "segments": [],
  "totals": {
    "distance_meters": 326000,
    "duration_seconds": 19200,
    "tolls_yuan": 148
  },
  "polyline": [],
  "fetched_at": "2026-07-10T10:00:00+08:00"
}
```

`status` 为 `provider`、`estimate` 或 `unavailable`。`provider` 只表示来自当前高德道路请求，不暗示实时交通。未来导航导出只依赖有序节点、坐标和路线几何，不依赖当前 UI 内部结构。

## 6. 约束语义

### 6.1 硬约束

- `required`：节点不可被优化器删除或移出行程。
- `fixed_day`：节点必须保留在当前日期。
- `fixed_time`：节点必须保持当前时段或用户定义的时间窗。
- `fixed_order`：节点作为当前作用范围中的顺序锚点。
- `CityStop.fixed_order`：城市作为全行程优化中的顺序锚点。
- 环线起终点：起点和终点为同一节点，均不可移动。
- 单程端点：指定起点和终点不可移动，中间节点可优化。

硬约束冲突时，优化接口返回结构化冲突列表，不返回一个看似成功的候选方案，也不自动放宽约束。

### 6.2 软目标

- 减少折返、道路距离和旅行时间。
- 控制每天驾驶和游览强度。
- 尽量遵守地点开放时间。
- 保留用户拖动后的偏好顺序。
- 提高景点价值、兴趣匹配和类型多样性。
- 避免跨城交通或固定时段冲突。

开放时间等外部数据可能不完整，因此默认作为软目标和警告。只有用户显式固定的日期或时段才升级为硬约束。

## 7. 优化流程

### 7.1 通用流程

1. 确认优化范围：某一天、某城市或整条路线。
2. 校验草稿 schema、revision、节点身份和硬约束。
3. 解析地点坐标；未定位节点退出本次计算并进入警告列表。
4. 获取所需的路网成本；外部响应必须解析和归一化。
5. 先放置固定日期、固定时段和固定顺序节点。
6. 只在锁定锚点之间排列未锁定节点。
7. 计算按天容量、时间冲突和路线指标。
8. 生成 `CandidatePlan`、差异摘要、数据来源和警告。
9. 浏览器再次验证 base revision 和硬约束，然后展示候选差异。
10. 用户选择应用或放弃。应用后仍保留一次可撤销的前一版本。

### 7.2 普通行程

普通行程优先保持日期和已有人工顺序。在选定作用范围内，使用地点坐标、预计停留时长、开放时间、当天可用时间和用户节奏安排未锁定节点。

算法使用确定性的初始排序和局部改进，不依赖 AI 计算地理顺序。AI 可以提供景点价值、兴趣匹配、取舍建议和自然语言解释，但输出先被归一化为软评分，再进入约束优化器。

### 7.3 自驾路线

自驾排序基于高德 Web Service 返回的真实道路成本。对常见的小规模节点集合，使用以下可解释流程：

1. 固定环线或单程端点。
2. 固定显式顺序锚点。
3. 在锚点之间使用最近邻构造初始路线。
4. 使用受约束的局部交换或 2-opt 消除明显折返。
5. 根据固定日期和每日驾驶上限拆分到各天。
6. 重新请求最终有序路线的分段道路详情。

三种策略只改变软目标权重：

- 少开车：道路时间和距离优先。
- 均衡：道路成本、每天负载、景点价值和停留时间综合权衡。
- 多体验：在每日驾驶上限内提高景点价值和多样性，允许有限绕行。

相同输入、相同外部数据和相同策略必须产生稳定结果，便于测试和解释。

自驾优化请求超过 20 个已定位节点时返回 422 和稳定错误码 `route_node_limit_exceeded`，不截断节点，也不在用户不知情时分批优化。

## 8. API 设计

### 8.1 保持现有接口

- `POST /api/plan` 保持现有用途和兼容性，继续生成初始规划。
- `GET /api/search_pois` 继续用于地点搜索。
- 现有交通、系统和位置接口不改变响应契约。

### 8.2 优化接口

新增 `POST /api/plan/optimize`：

```json
{
  "base_revision": 12,
  "scope": { "type": "day", "id": "day-2" },
  "draft": {}
}
```

`scope.type` 为 `day`、`city` 或 `trip`。策略和模式读取自 draft，避免同一请求存在两个互相冲突的来源。

成功响应：

```json
{
  "base_revision": 12,
  "candidate": {},
  "diff": [],
  "warnings": [],
  "data_sources": []
}
```

`diff` 中每一项使用 `add`、`move`、`remove` 或 `update` 类型，包含稳定 `node_id`、变更前位置、变更后位置和面向用户的原因。约束冲突使用 422 响应，错误码为 `constraint_conflict`，并返回冲突约束、节点 ID 和日期 ID。第三方服务不可用使用可识别的降级状态，不泄露供应商响应、密钥或堆栈。

### 8.3 自驾道路接口

新增 `POST /api/transport/driving-route`。它按照请求中的当前节点顺序计算路线，不做 AI 取舍，也不擅自重排节点：

```json
{
  "route_shape": "round_trip",
  "nodes": []
}
```

响应包含 `source`、`status`、`ordered_node_ids`、`segments`、`totals`、`polyline` 和 `warnings`。这个 provider-neutral 契约是未来导航导出的预留接口。

### 8.4 地图点选解析

新增 `GET /api/reverse_geocode?lat=<latitude>&lng=<longitude>`，用经纬度执行反向地理编码并返回经过归一化的地点候选。该接口只负责解析地点，不写入服务器状态。

## 9. 代码边界

### 9.1 前端

- `static/js/core/state.js`：扩展初始状态和 schema 版本，不承载编辑算法。
- 新的纯工具模块：草稿转换、约束校验、不可变编辑操作和 revision 处理。
- 新的编辑器模块：想去清单、每日节点、自驾节点和候选差异渲染。
- `static/js/planning/history.js`：有界撤销/重做历史。
- `static/js/core/storage.js`：保存 version 2 快照并迁移 version 1 数据。
- `static/js/planning/map.js`：渲染已安排、未安排和道路路线三类视觉状态。
- `static/js/app.js`：只保留启动、事件绑定、状态协调和模块调用。

所有模块继续通过 `window.AeroTravel*` 命名空间暴露，并在 `static/index.html` 中于 `static/js/app.js` 之前加载。不引入框架或构建工具。

### 9.2 后端

- `schemas/`：新增草稿、节点、约束、优化请求和道路请求模型。
- `clients/amap.py`：增加反向地理编码和道路数据的底层调用与响应解析。
- `services/`：增加无状态道路成本和最终路线服务。
- `planner/constraints.py`：纯硬约束校验与冲突报告。
- `planner/route_optimizer.py`：确定性路线排序和按天拆分。
- `planner/draft_optimizer.py`：协调普通行程和自驾候选生成。
- `routers/planning.py`：注册优化接口。
- `routers/transport.py`：注册自驾道路接口。
- `routers/location.py`：注册地图点选解析接口。

`server.py` 仍然只负责应用组合，不回收业务逻辑。

## 10. 持久化与迁移

本地快照增加 `schema_version`。读取 version 1 快照时：

- 先走现有 `applyPlan()` 还原行程。
- 为每个现有 item 生成稳定节点 ID。
- 将现有日程转换为 `AppliedPlan` 和等价 `WorkingDraft`。
- 想去清单为空，所有新增约束默认为 false。
- 迁移成功后只有在用户再次保存时才写入 version 2，不覆盖原始快照。

迁移失败时显示该快照无法编辑，但仍尝试使用旧的只读恢复路径。不得删除或静默覆盖用户历史。

version 1 迁移 ID 由快照标识、日期、原数组位置、地点名称和坐标的规范化值确定性生成。同一个旧快照重复读取时必须得到相同 ID；随机 ID 只能用于用户新建的节点。

撤销历史只需保留当前编辑会话中的最近 50 个操作，不写入长期快照。快照继续遵守现有最多 8 条限制。

## 11. 异常与降级

| 场景 | 产品行为 |
|---|---|
| 地点无法定位 | 保留在想去清单并标记待定位，不参与优化 |
| 硬约束冲突 | 阻止候选生成，列出冲突节点和约束 |
| 高德道路接口失败 | 保留草稿；仅保存仍可用；估算必须标记为估算 |
| 部分道路分段失败 | 标记失败分段，不给出伪造的总费用或总时间 |
| AI 请求失败 | 手工编辑、约束、撤销和道路计算继续可用 |
| 优化响应 revision 过期 | 静默丢弃响应，并提示草稿已更新 |
| 用户取消优化 | 通过 `AbortController` 中止前端等待，草稿不变 |
| localStorage 写入失败 | 保留内存状态并明确提示本次修改未持久化 |
| version 1 快照迁移失败 | 使用只读恢复，不覆盖旧数据 |

所有外部数据和错误文本继续按不可信输入处理。日志使用 `core.observability.log_event()`，只记录错误类型、请求 ID、作用范围和节点数量，不记录密钥、完整用户偏好或供应商原始响应。

## 12. 可访问性与移动端

- 拖拽不是唯一操作方式；每个节点提供移动到某天、上移和下移命令。
- 拖动手柄具备可访问名称，键盘可以聚焦节点操作菜单。
- 锁定使用图标、文本和状态属性共同表达，不能只依赖颜色。
- 候选差异按新增、移动、移除、冲突分类，屏幕阅读器可以按顺序读取。
- 移动端不同时展示想去清单和日程双栏，使用两个页签或抽屉切换。
- 地图不是完成编辑的必需入口，所有地点和路线操作均可从列表完成。

## 13. 分阶段交付

### 阶段 1：可编辑行程核心

- version 2 草稿模型和 version 1 快照迁移。
- 想去清单和三种地点添加入口。
- 同日排序、跨天移动、删除和四类约束。
- 仅保存、撤销/重做和桌面/移动端编辑界面。

### 阶段 2：候选与局部优化

- 硬约束校验和结构化冲突。
- 按日、按城市和全行程作用范围。
- 候选版本、差异预览、应用和撤销。
- AI 软评分与确定性地理排序分离。

### 阶段 3：自驾规划

- 环线和单程路线编辑。
- 高德道路分段、里程、时间、过路费和路线几何。
- 三种优化策略和受约束的路线改进。
- 按天路线拆分和驾驶强度检查。

### 阶段 4：导航适配预留验证

- 验证有序途经点和路线几何足以支撑后续导航导出。
- 不在本阶段实现逐向导航或第三方导航跳转。

每个阶段必须可以独立通过 `scripts/check.ps1`，并保持前一阶段可用，不以未完成的后续阶段作为运行前提。

- 环线、单程、固定端点和固定顺序锚点。
- 最近邻初始解和受约束 2-opt 的稳定结果。
- 三种策略的固定输入结果。
- 按天拆分和每日驾驶上限。
- 高德道路响应、过路费和 polyline 解析。
- provider、估算和不可用三种来源状态。

所有测试使用固定 fixture 和 mock，不依赖在线高德或 AI 服务。

### 14.3 路由测试

- 保持 `/api/plan` 兼容。
- 优化成功、硬约束 422、过期 revision 和第三方降级。
- 自驾道路接口保持输入顺序，不做隐式重排。
- 反向地理编码返回归一化地点，不泄露供应商原始错误。

- 页面启动、初始示例、地图和控制台无错误或警告。
- 搜索、地图点选和名称粘贴均能加入想去清单。
- 仅保存不会改变其他节点。
- 智能优化只影响所选范围内未锁定节点。
- 候选方案可以查看、放弃、应用和撤销。
- 自驾节点拖动后路线指标更新，不调用 AI。

- 刷新页面后恢复想去清单、约束和已应用结果。
- 桌面和移动端均能在不使用拖拽的情况下完成编辑。

## 15. 验收标准

1. 用户向已生成行程加入一个景点并选择仅保存时，其他节点的日期、顺序和内容完全不变。
2. 用户主动加入的地点默认必去，优化器不得删除；改为可选后才允许在候选方案中建议移除。
3. 用户选择优化某一天时，其他日期不发生变化。
4. 优化器不得删除必去节点，不得移动固定日期、固定时段或固定顺序节点。
5. 冲突的硬约束必须阻止优化并给出具体节点，不得自动妥协。
6. 手动拖动自驾节点后，无需 AI 即可获得更新后的真实道路指标或明确的降级状态。
7. 环线保持同一起终点，单程保持指定起点和终点。
8. 未定位地点可保存但不能进入路线计算，界面始终显示其状态。
9. 候选结果应用前可查看差异，应用后至少可以撤销一次。
10. AI、高德或 localStorage 任一失败都不能破坏当前已应用行程。
11. 旧快照可以只读恢复，并在迁移成功后继续编辑。
12. `.\scripts\check.ps1` 通过，新增行为具有离线回归测试。
13. 浏览器烟测覆盖桌面和移动视口，地图、初始行程和编辑控制均正常。

## 16. 设计结论

本设计采用约束驱动的混合规划：AI 负责生成初稿、景点价值和解释；高德负责可信地点与道路数据；确定性优化器负责在硬约束内排序；用户始终决定何时保存或应用候选方案。

普通行程和自驾路线共享同一套草稿、稳定节点、约束、候选差异、revision 和撤销机制。这样可以先解决生成后不可编辑的问题，再在同一基础上增加自驾，而不会形成两套彼此冲突的产品和代码路径。
