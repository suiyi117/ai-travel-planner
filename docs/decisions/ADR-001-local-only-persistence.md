# ADR-001: 保持本地快照持久化，暂不引入登录与服务端数据库

## Status

Accepted

## Date

2026-07-09

## Context

AeroTravel 当前定位是单用户旅行规划工具。已有约束是无数据库、无构建步骤、无登录系统，前端将最近生成的行程快照保存在浏览器 `localStorage` 的 `aerotravel:trips` 中。后端主要负责 AI 行程生成、高德代理、天气、12306/航班参考数据。

产品化阶段需要明确持久化和 auth 范围，否则后续会在三个方向之间摇摆：

- 继续本地-only 快照；
- 增加可分享快照链接；
- 增加账号、云端数据库和跨设备同步。

## Decision

当前版本继续采用本地-only 快照持久化，暂不引入登录系统、用户账号、服务端数据库或云端同步。

## Rationale

- 当前项目是单用户工具，本地快照已经覆盖“生成后可恢复、可带出门、可导出日历”的核心需求。
- 引入 auth 和数据库会带来用户数据安全、密钥管理、隐私合规、备份恢复、迁移和运维成本。
- 后端目前没有稳定的用户身份边界；贸然增加账号会改变 API、安全模型和部署复杂度。
- 保持本地-only 可以继续维持无数据库、无构建步骤和低部署成本。

## Alternatives Considered

### Shareable Snapshot Links

将行程 JSON 上传到后端或对象存储，生成只读分享链接。

- Pros: 可分享、可跨设备打开。
- Cons: 需要服务端持久化、过期策略、删除机制和隐私提示。
- Decision: 暂不实现，作为未来独立功能评估。

### Authenticated Cloud Persistence

增加登录、用户表、行程表和跨设备同步。

- Pros: 体验完整，支持多设备和长期保存。
- Cons: 引入数据库、认证、权限、数据导出/删除、备份、监控和更高安全责任。
- Decision: 当前阶段不符合项目规模和部署约束。

### Local File Export Only

完全移除 localStorage，只保留 `.ics` 或 JSON 文件导出。

- Pros: 服务端零状态，隐私清晰。
- Cons: 用户恢复体验差，刷新页面后丢失行程。
- Decision: 不采用，保留 localStorage 快照。

## Consequences

- 行程快照只存在于当前浏览器和当前设备。
- 清理浏览器数据、隐私模式或禁用存储会导致历史快照不可用。
- 服务端无需处理用户数据存储、账号权限和数据库迁移。
- 后续若要做分享或云端同步，必须先写新的 ADR，并补充隐私、删除、过期和迁移策略。

## Review Trigger

当出现以下需求之一时重新评估：

- 明确要求跨设备同步；
- 明确要求分享链接；
- 需要团队协作或多人共同编辑行程；
- 需要长期保存用户历史行程；
- 准备面向公开用户大规模发布。
