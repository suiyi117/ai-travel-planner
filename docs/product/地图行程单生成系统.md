该笔记展示了作者经过多轮沟通后，使用 GPT 生成一张集成每日时间表、景区路线、徒步路线、返程、预算和提示的完整行程图。它证明了“一张图建立全局认知”具有很强的视觉吸引力和传播性。

该案例也同时暴露了生成式图片的风险：作者明确提到图中存在错别字，评论区出现了“好看不代表好用”和“建议多检查”的反馈。因此 AeroTravel 只参考其信息组织、地图呈现和视觉表达，不直接复制文案、图片或视频，也不使用生成式 AI 图片作为可执行行程的事实来源。

对 AeroTravel 的直接产品启示是：允许 AI 帮助提出版式和内容建议，但地点、坐标、路线、时间、费用和最终文字必须来自结构化数据并通过规则与人工复核。小红书的 `xsec_token` 可能随时效或登录状态变化，后续如无法打开，仍以笔记 ID 作为参考索引。

## 问题陈述

如何让客户在出发前的 10 秒内，看懂整趟旅行的地点顺序、每日区域、交通衔接和重要提示，同时保留旅途中查看详情、打开导航和离线保存的能力？

## 格式决策

不在 HTML、PNG、PDF 和 SVG 之间选择单一格式，而是由同一份结构化行程数据生成三种客户成品：

1. **专属 HTML 页面**：主交付物，负责地图互动、按天查看、下一站和外部导航。
2. **PNG 总览图**：负责让客户快速建立全局认知，方便在私信、相册和小红书中查看。
3. **PDF 行程单**：负责完整留档、打印、离线阅读和保留可点击链接。

SVG 只作为系统内部的路线、箭头、编号、标签和图例绘制层，不作为普通客户的直接交付文件。生成式 AI 图片只能用于封面或装饰，不能承担地理位置、路线、时间或费用等事实信息。

## 单一数据源

```text
AeroTravel 结构化行程
  |
  +-- 城市与每日顺序
  +-- 地点名称、坐标、时间和停留时长
  +-- 交通段、距离、耗时和数据可信度
  +-- 真实道路 polyline 或明确标记的估算连线
  +-- 预算、天气、提示和外部参考链接
  |
  v
统一行程交付渲染器
  |
  +-- HTML 互动页
  +-- PNG 总览图
  +-- PDF 行程单
```

任何人工修改都先进入结构化行程，再重新生成三种交付物。不允许分别手改 HTML、PNG 和 PDF，避免客户收到互相矛盾的版本。

## 信息层级

总览页只回答“整趟旅行怎么走”，不承担完整攻略。信息优先级从高到低为：

1. 行程名称、日期、天数和地点组合。
2. 总体路线、城市顺序、住宿点和交通枢纽。
3. 每天主要区域和核心任务。
4. 城际与当日关键交通方式。
5. 总预算、重要风险和出发前必做事项。
6. 二维码、更新时间和信息声明。

景点长描述、餐厅口碑、电话、开放时间和备选方案下沉到每日详情页，不挤占总览地图。总览每日摘要优先展示核心景点名（而非仅城级游玩句）；静态总览图优先道路 polyline，标记跨日均衡选取核心锚点。

## 第一版桌面与 A4 横向版式

用于桌面 HTML 总览、客户完整 PNG 和 A4 横向 PDF。地图是第一视觉主体，时间表和提示为辅助信息。

```text
+-----------------------------------------------------------------------+
| 行程名称                       日期 / 天数 / 预算 / 更新时间 |
+----------------------+--------------------------------+---------------+
| 每日摘要             |                                | 关键数据      |
|                      |                                |               |
| D1  城市 / 区域      |          总体路线地图          | 交通预算      |
|     核心安排         |                                | 餐饮预算      |
|                      |   [1] ---- [2] ==== [3]         | 总预算        |
| D2  城市 / 区域      |    |             |              |               |
|     核心安排         |   [4] ---------- [5]            | 重要提示      |
|                      |                                |               |
| D3  城市 / 区域      |   住宿 / 车站 / 机场 / 日区域标识 | 天气 / 预约   |
|     核心安排         |                                | 风险 / 备案   |
+----------------------+--------------------------------+---------------+
| 图例 / 交通可信度 / 信息声明           扫码打开专属行程 [QR] |
+-----------------------------------------------------------------------+
```

### 宽度比例

- 每日摘要区约占 22%。
- 总体地图区约占 56%。
- 预算与提示区约占 22%。
- 地图高度不低于可见内容的 60%。
- 顶部与底部只承担全局信息，不放大段文字。

## 第一版手机 HTML 版式

手机上不缩小桌面行程单，而是改为单列滚动。首屏必须先给客户全局感，然后再进入每日详情。

```text
+------------------------------+
| 行程名称                     |
| 城市顺序 / 日期 / 天数       |
+------------------------------+
|                              |
|         总体路线地图           |
|   [1] ---- [2] ==== [3]      |
|             |                |
|            [4]               |
|                              |
+------------------------------+
| 今日 / 全程切换               |
+------------------------------+
| D1  城市·主区域   三个关键点    |
| D2  城市·主区域   三个关键点    |
| D3  城市·主区域   三个关键点    |
+------------------------------+
| 总预算 / 天气 / 两条最重要提示  |
+------------------------------+
| [查看每日详情] [下载 PDF]    |
+------------------------------+
```

### 手机交互

- 地图支持缩放和拖动，但首次打开时自动适配全部路线。
- 点击编号后，底部显示地点名称、时间和“高德打开”。
- 地图下方使用日期标签或紧凑的分段控件切换“全程 / 今日”。
- 每日摘要一行最多保留三个路线锚点，完整时间表进入下级详情。
- 主要操作是“查看每日详情”，不在总览层堆叠过多按钮。

## PNG 总览图版式

首版优先生成竖向 `3:4` 总览图，适合手机查看和小红书分页。它不是完整时间表，只包含：

- 行程名称和城市顺序。
- 占画面约 55% 的总体地图。
- 每天一句话摘要。
- 总预算和两条重要提示。
- 专属链接二维码。
- 更新时间与简短声明。

桌面横向 PNG 可作为第二导出模板，但不是手机私信的默认附件。

## PDF 行程单版式

- 第 1 页：A4 横向总览，使用“每日摘要 / 总体地图 / 预算提示”三栏。
- 第 2 页起：每天一页或两天一页，根据信息密度自动选择。
- 每日页包含当日地图、时间线、交通、备选和风险提示。
- 外部参考和预约使用可点击文字链接，不打印超长 URL。
- PDF 中的文字保持为可选择文本，不将整页压成一张低清图片。

## 地图视觉规则

### 编号与层级

- 总览地图只标记城市、住宿、交通枢纽和每日核心锚点。
- 普通行程结果每天最多展示 5 个编号锚点，其余地点进入详情。
- 地点编号与每日摘要的编号必须一致。
- 住宿、机场、车站使用图标和文字，不混入景点数字序列。

### 路线类型

- 城际交通：粗实线 + 火车、飞机或汽车图标。
- 驾车真实道路：实线 + “道路数据”标识。
- 步行或游玩：细实线 + 步行图标。
- 估算路线：虚线 + “估算”文字，不仅依赖颜色区分。
- 延程建议：点线 + “可选”文字。

### 色彩

- 每天使用稳定日颜色，但同时显示 `D1`、`D2` 等文字，不只用颜色表意。
- 地图底图使用低饱和、低对比度样式，路线和编号标记保持最高对比度。
- 默认日颜色不超过 5 组，超过 5 天后按周期复用颜色并依赖日编号。
- 不使用大面积紫色渐变、高饱和度底图或大量装饰性阴影。

### 照片

- 第一版总览地图不放景点圆形照片，避免遮挡地理信息并减少版权风险。
- 可在行程封面使用一张有授权的目的地主图。
- 有可验证的照片授权和稳定来源后，再评估地图照片标注。

## 生成技术方向

### HTML 互动页

- 继续使用 Leaflet 和当前高德瓦片边界。
- 路线优先使用 AeroTravel 已有的真实 polyline。
- 数据不可用时，使用虚线估算并明确标识，不伪装成真实道路。
- 地点、预算、描述和外部资料继续进行 HTML 转义。

### PNG 导出

- 使用高德官方静态地图作为可导出的真实底图。
- 使用 SVG 或可控的 HTML 覆盖层绘制路线、编号、图例和文字。
- 再将组合结果栅格化为 PNG，不使用生成式 AI 重画地图。
- 高德静态地图官方能力限制为单图最大 `1024 x 1024`、最多 10 个标记和 4 条路径。超出后使用概览聚合或按天拆图。

### PDF 导出

- 使用专门的打印样式复用 HTML 结构。
- 地图使用已生成的静态图片，避免打印时在线瓦片缺失。
- 文字、表格和链接保持为 HTML/PDF 文本层，确保可搜索、可选择和可点击。

官方参考：[高德静态地图 API](https://lbs.amap.com/api/webservice/guide/api/staticmaps)、[SVG 技术参考](https://developer.mozilla.org/en-US/docs/Web/SVG)。

## 与当前代码的衔接

AeroTravel 已有的以下能力可以直接复用：

- `services/driving_route_service.py`：真实高德道路段、距离、耗时、费用与 polyline。
- `schemas/draft.py`：地点坐标、每日顺序、自驾路线形状和路线响应结构。
- `static/js/planning/map.js`：Leaflet 地图初始化和真实/估算路线样式。
- `static/js/delivery/delivery.js` 与 `static/css/delivery.css`：现有文字和长图交付边界。
- `static/js/app.js`：当前行程状态、地图点位和导出编排。

后续实现时，地图行程单应作为独立的无构建模块，不将大量生成与排版逻辑放回 `static/js/app.js`。

## 验收标准

- [ ] 客户在 10 秒内能指出城市或主要地点的访问顺序。
- [ ] 首屏能看到行程名称、总体地图和城市顺序。
- [ ] 手机 `320px` 宽度下无文字重叠、溢出和不可点击控件。
- [ ] 桌面、竖向 PNG 和 A4 横向 PDF 使用同一份行程数据。
- [ ] 地图编号与每日摘要顺序一致。
- [ ] 真实道路和估算连线同时通过线型和文字标识区分。
- [ ] 所有地点名、描述、预算和外部资料在 HTML 插入前均被转义。
- [ ] 二维码与文字链接都能打开对应的专属行程页。
- [ ] PDF 文字可选择，外部链接可点击，地图在断网时仍能显示。
- [ ] 地图、图例、标记和页面文字不发生不可读的覆盖。

## 暂时不做

- 不用生成式 AI 画真实地图或路线。
- 不把参考图的所有文字原样塞进手机首屏。
- 不在第一版总览地图上使用景点圆形照片标注。
- 不为 PNG、PDF 和 HTML 维护三份独立内容。
- 不在一张总览地图上展示超过 10 个细节标记。
- 不在没有官方路线数据时将两点连线伪装成真实道路。
- 不在总览页放入完整餐厅评价、所有备选点和大段声明。

## 开放问题
