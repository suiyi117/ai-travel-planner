# AeroTravel 路线设置交互优化 PRD

> 状态：Proposed，待排期  
> 日期：2026-07-15  
> 版本：v1.0  
> 优先级：P0  
> 范围：三步向导 Step 1「路线」，以及与 Step 2、Step 3、行程快照相连的必要状态  
> 关联设计：[页面信息架构重设计](../superpowers/specs/2026-07-11-page-redesign-wizard-design.md)  
> 关联 PRD：[自驾规划交互重构](./自驾规划交互重构-PRD.md)

## 1. 结论摘要

当前页面的视觉方向和三步向导结构可以保留，不需要推翻重做。现有实现已经具备城市添加、停留/过境、排序、段级交通、单程/环线、日期、天数汇总和步骤校验等核心能力。

本轮真正需要补齐的是“规则可见性”和“边界反馈”：

1. 明确第一座城市是出发地，后续城市是途经或游玩目的地。
2. 明确城市卡中的交通字段表示“从上一站到达本城市”的方式。
3. 让空输入、重复城市、无效天数、全程过境、删除失败等操作都有即时反馈。
4. 明确环线、排序、删除会怎样影响起点、回程和交通段。
5. 明确“计划天数”只汇总游玩天数，不自动增加独立转场日。
6. 让长路线下的总览与“下一步”始终容易找到。
7. 补齐键盘、触屏、窄屏、旧快照和重新生成时的状态规则。

完成本 PRD 后，Step 1 应从“能填写的城市列表”升级为“规则清楚、结果可预测的路线编辑器”。

## 2. 背景与问题

### 2.1 当前能力

当前 Step 1 已实现：

- 按输入顺序添加城市；
- 第一座城市默认作为出发/过境起点；
- 每座城市可选择游玩或过境；
- 游玩城市可设置 1–7 天；
- 第二座城市起可设置到达方式；
- 支持拖拽、上移、下移和删除；
- 支持单程与环线；
- 环线生成时补充最后一城返回起点的交通；
- 总天数由各城市游玩天数求和；
- 至少存在一个游玩城市后才可进入 Step 2；
- 已生成行程后返回修改，可通过设置差异触发重新生成。

### 2.2 当前主要缺口

| 编号 | 缺口 | 用户风险 |
|---|---|---|
| G-01 | “目的地城市序列”包含出发地，命名与数据语义不完全一致 | 不知道第一城是否要游玩、是否算目的地 |
| G-02 | “到达：自驾优先”挂在城市卡上，缺少上一站上下文 | 不知道交通属于哪一段 |
| G-03 | 空输入、重复城市和唯一城市删除会静默无效 | 用户误以为按钮失灵 |
| G-04 | 城市名接受任意文本，且无格式或长度限制 | 错误名称进入 POI、天气和 AI 请求 |
| G-05 | 单城市可选择环线，但前端摘要和后端交通段语义不一致 | 可能显示“北京 → 北京”但没有真实回程段 |
| G-06 | 排序后起点与交通段会改变，但页面没有结果提示 | 用户可能无意中改变整条路线 |
| G-07 | 删除、排序、改变环线后没有清晰说明旧结果已过期 | 用户可能把旧行程当成新路线结果 |
| G-08 | “计划天数”容易被理解为包含转场的总行程天数 | 生成结果与用户预期相差一天 |
| G-09 | 长城市列表下，摘要和下一步按钮离当前操作较远 | 不知道是否填完，操作成本高 |
| G-10 | 拖拽手柄和图标操作的触屏、键盘与焦点规则不完整 | 无障碍和移动端体验不稳定 |
| G-11 | 出发日期缺省和过去日期的处理没有用户可见规则 | 班次日期、天气和导出日期可能不符合预期 |
| G-12 | 交通方式随“到达城市”保存，排序后的继承规则未显式说明 | 重排后段级交通可能与原意不一致 |

## 3. 产品目标

### 3.1 目标

- 用户不阅读帮助文档也能说清起点、停留城市、过境城市和回程的关系。
- 用户始终知道交通方式作用于哪一段。
- 每次无效操作都有就近、可理解、可恢复的反馈。
- 路线顺序、总游玩天数、路线形态和下一步状态实时可见。
- 修改路线后，已生成行程是否需要重新生成有明确状态。
- 不改变现有后端规划契约、单一 `state`、`applyPlan()` 和 localStorage 快照模式。

### 3.2 成功标准

- 首次用户可在 2 分钟内完成“出发地 + 2 个目的地 + 天数 + 环线”设置。
- 用户能准确回答某个交通选择对应的起止城市。
- 空输入、重复输入和全程过境均不会出现“无反应”。
- 城市重排后，路线摘要、起点标签、到达段和总天数立即一致。
- 长路线下无需滚到页面最底部也能看到当前摘要和主操作。
- 桌面、窄屏和键盘操作均可完成 Step 1。
- `scripts/check.ps1` 和浏览器冒烟全部通过。

## 4. 非目标

本次不包含：

- 新增数据库、账号、云同步或多人协作；
- 引入 React、Vue、构建工具或新的状态管理库；
- 恢复常驻地图或锁死 `100vh` 的三栏布局；
- 重写 AI 规划、12306、航班或驾车路线算法；
- 修改运输段稳定字符串格式 `"A → B"`；
- 增加必须付费的城市联想服务；
- 将交通转场自动扩展为独立行程日；
- 在本轮重做 Step 2 偏好和 Step 3 时间线视觉；
- 改变每城 1–7 天的现有 Step 1 产品限制。

## 5. 用户与核心场景

### 5.1 异地出发，多城游玩

用户从淮北出发，不在淮北游玩，计划合肥 1 天、武汉 2 天，最后返回淮北。

期望路线：

```text
淮北（出发/过境） → 合肥（游玩 1 天） → 武汉（游玩 2 天） → 淮北（自动回程）
```

### 5.2 起点也安排游玩

用户从成都开始川西自驾，第一天先在成都游玩，再前往康定。

期望：成都卡片标记“起点 · 安排游玩”，成都天数计入总游玩天数。

### 5.3 中间城市仅过境

用户从上海前往黄山，中间在杭州换乘但不游玩。

期望：杭州保留在路线和交通段中，但不生成杭州游玩日，也不计入游玩天数。

### 5.4 修改已生成路线

用户已生成“北京 → 西安”，返回 Step 1 增加洛阳并排序。

期望：旧行程继续保留用于返回查看，但页面明确显示“路线已修改，需要重新生成”；用户确认后才覆盖正式行程。

## 6. 术语与数据语义

### 6.1 用户可见术语

| 术语 | 定义 |
|---|---|
| 出发地 | 城市序列中的第一座城市，也是环线回程终点 |
| 游玩城市 | `plan_stay=true` 且 `days>=1`，生成按日游玩内容 |
| 过境城市 | `plan_stay=false` 且 `days=0`，保留路线与交通，不生成游玩日 |
| 到达方式 | 从上一座城市前往当前城市的段级交通偏好 |
| 单程 | 行程结束于列表最后一座城市 |
| 环线 | 行程结束后从最后一座城市返回第一座城市 |
| 游玩天数 | 所有城市 `days` 之和，不额外计算独立转场日 |

### 6.2 数据字段映射

| UI 概念 | 当前字段 | 规则 |
|---|---|---|
| 城市顺序 | `state.cities[]` | 数组顺序即路线顺序 |
| 城市名称 | `city.name` | 去除首尾空格后保存，渲染前转义 |
| 是否游玩 | `city.plan_stay` | `false` 时强制 `days=0` |
| 游玩天数 | `city.days` | 游玩城市 1–7；过境城市为 0 |
| 到达方式 | `city.transport` | 第 2 城起表示“上一城 → 当前城” |
| 路线形态 | `state.routeShape` | `one_way` 或 `round_trip` |
| 出发日期 | `departureDate` / `start_date` | 默认次日；用于班次、天气和导出 |

## 7. 信息架构

### 7.1 页面保持三步向导

```text
路线 → 偏好 → 行程
```

- Step 1/2 保持自然页面滚动；
- Step 3 继续使用工作区和按需地图抽屉；
- 不恢复常驻大地图；
- 不把生成按钮放到顶栏。

### 7.2 Step 1 推荐结构

```text
页面标题与说明
城市输入
路线形态
路线编辑列表
出发日期
游玩天数汇总
步骤操作区
```

桌面宽屏保留项目既定的粘性摘要能力；若当前视觉版本采用居中单列，至少必须提供粘性底部操作条，不能让摘要和“下一步”同时离开视口。窄屏只保留顶部步骤条、紧凑摘要和底部主操作。

## 8. 详细功能需求

### FR-01 城市输入与添加

1. 输入框标签改为“城市路线”，帮助文案说明“第一座城市将作为出发地”。
2. 输入值保存前执行：
   - 去除首尾空格；
   - 连续空白合并为一个空格；
   - 拒绝纯空白；
   - 最大长度 30 个字符；
   - 拒绝仅由标点或控制字符组成的名称。
3. 点击“添加”或按 Enter 使用同一条添加逻辑。
4. 空输入时不清空输入框，在输入框下显示“请输入城市名称”。
5. 重复判断至少使用规范化后的完全匹配；重复时聚焦并短暂高亮已存在卡片，提示“该城市已在路线中”。
6. 添加成功后：
   - 清空输入框；
   - 新卡片显示轻量入场反馈；
   - 路线摘要、总天数、步骤校验立即更新；
   - 焦点回到输入框，便于连续添加。
7. 不强依赖在线城市校验。高德不可用时仍允许添加，后续 POI 和地图使用现有 fallback。
8. 路线最多包含 10 座城市（包括出发地和过境城市）。达到上限后禁用添加按钮，并提示“单次最多规划 10 座城市”。该上限与当前静态地图点位展示能力保持一致。
9. P1 可增加最近使用城市或本地常见城市建议，但不得新增必需 API Key。

### FR-02 第一座城市与起点规则

1. 数组索引 0 永远是当前出发地。
2. 第一张卡片必须显示“出发地”高识别标签。
3. 第一城默认：`plan_stay=false`、`days=0`、不显示到达方式。
4. 用户勾选“在出发地游玩”后，天数默认变为 1，可改为 1–7。
5. 第一城被删除或移到其他位置后，新索引 0 自动成为出发地。
6. 起点变化时显示非阻断提示：
   - 单程：“出发地已改为 {城市}。”
   - 环线：“出发地和回程终点已改为 {城市}。”

### FR-03 游玩与过境

1. 每城提供一个明确的二态控制：“安排游玩 / 仅途经”。可继续使用复选框，但标签必须完整表达结果。
2. 切换为“安排游玩”：
   - `plan_stay=true`；
   - 若原天数为 0，恢复为 1；
   - 天数控件可用。
3. 切换为“仅途经”：
   - `plan_stay=false`；
   - `days=0`；
   - 天数控件禁用或折叠；
   - 城市继续保留在路线和交通段中。
4. 至少需要一个游玩城市。全程过境时禁止进入 Step 2，并提示“请至少为一座城市安排游玩天数”。
5. 过境城市不得请求 POI 游玩数据，不生成该城市 Day 内容。

### FR-04 游玩天数

1. 游玩城市允许 1–7 天；过境城市固定为 0 天。
2. 修改天数后即时更新总游玩天数和摘要，不发起 AI 请求。
3. 页面名称统一为“游玩天数”，避免写成可能包含交通日的“总行程天数”。
4. 汇总文案：

   ```text
   共 3 个游玩日 · 2 个游玩城市 · 1 个过境城市
   ```

5. 帮助文案明确：“城际转场默认安排在相邻游玩日内，不自动增加独立转场日。”
6. 全程游玩天数允许 1–30 天。超过 30 天时保留用户已填数据，但禁止进入 Step 2，并提示拆分为多次规划。
7. 若未来支持独立转场日，必须作为新的领域字段和单独 PRD，不通过隐式加一天实现。

### FR-05 到达方式与交通段

1. 第一城不显示到达方式。
2. 第 `n` 个城市的 `transport` 永远表示：

   ```text
   cities[n-1].name → cities[n].name
   ```

3. UI 必须显示完整上下文，例如“从淮北到达合肥：自驾优先”，或在卡片上显示“上一段：淮北 → 合肥”。
4. 选项保持：智能推荐、高铁优先、飞机优先、自驾优先。
5. 城市重新排序后：
   - `transport` 继续随到达城市移动；
   - 文案立即按新的上一站重算；
   - 显示提示“路线顺序已更新，请检查各段到达方式”；
   - 不静默猜测或自动重置用户选择。
6. 环线回程段默认沿用 Step 2 的全局交通偏好；若全局为智能推荐，沿用现有后端回退规则。本轮不新增回程段单独交通控件。
7. 所有运输段 key 继续使用稳定格式 `"A → B"`。

### FR-06 单程与环线

1. 单程：摘要停在最后一城。
2. 环线：摘要追加第一城，但不得在 `state.cities` 中复制起点。
3. 城市少于 2 座时禁用环线，并提示“添加至少两个城市后可设置环线”。
4. 从单程切换环线后提示“行程结束后将返回 {起点}”。
5. 从环线切换单程后，移除结果中的环线回程段；已生成行程进入待重新生成状态。
6. 排序导致起点改变时，回程终点同步改变。
7. 环线摘要、规划 Prompt、交通段和末日回程必须使用同一个 `routeShape` 来源。

### FR-07 城市排序

1. 桌面支持拖拽；触屏和键盘场景使用上移/下移按钮。
2. 拖拽仅从手柄启动，避免在卡片空白区选择文本时误拖。
3. 拖拽中显示原位置和目标插入位置。
4. 到达首尾边界时，对应移动按钮禁用，而不是点击无反应。
5. 排序完成后同步更新：
   - 序号；
   - 出发地标签；
   - 到达段文案；
   - 环线回程终点；
   - 路线摘要；
   - 已生成结果的脏状态。
6. 排序不改变城市的游玩/过境和天数设置。

### FR-08 删除城市

1. 删除按钮使用“移除 {城市}”的可访问名称，点击区域至少 44×44 CSS 像素。
2. 未生成行程时：
   - 删除普通城市立即生效；
   - 删除起点立即生效，并提示新的起点；
   - 仅剩一城时允许删除，进入空路线状态，不再静默阻止。
3. 已生成行程后删除城市不立即覆盖旧结果，只将设置标为已修改。
4. 删除不弹阻断确认框；删除后提供 5 秒“撤销”Toast。
5. 撤销恢复原索引、天数、过境状态和交通设置。

### FR-09 出发日期

1. 默认值为当前本地时区的次日。
2. 日期允许清空，清空后显示“日期未定”；生成仍可继续。
3. 选择过去日期时显示字段错误并禁止进入 Step 2。
4. 日期改变后更新班次查询、天气提示和导出基准日期；不立即请求 AI。
5. 已有结果时改变日期，将设置标记为已修改。
6. 所有日期计算使用用户本地日期，避免 UTC 转换导致跨日。

### FR-10 路线摘要

摘要至少包含：

- 完整或可展开的路线；
- 单程/环线；
- 游玩天数；
- 游玩城市数与过境城市数；
- 出发日期或“日期未定”；
- 校验状态。

示例：

```text
淮北 → 合肥 → 武汉 → 淮北
环线 · 3 个游玩日 · 2 个游玩城市 · 7月20日出发
```

桌面宽屏使用粘性摘要区域；窄屏使用可折叠紧凑摘要，不与页面内容重复堆叠。

### FR-11 下一步与步骤解锁

1. 主按钮文案保持“下一步：偏好 →”。
2. 长列表下主按钮应通过粘性底部操作条保持可达。
3. 按钮状态：
   - 路线为空：禁用；
   - 没有游玩城市：禁用；
   - 存在字段错误：禁用；
   - 校验通过：可用。
4. 禁用状态旁必须说明原因，不能只改变透明度。
5. 点击下一步时执行一次完整校验；首个错误滚入视图并聚焦。
6. Step 2 只有在 Step 1 校验通过后解锁；Step 3 只有存在已应用行程后解锁。

### FR-12 已生成行程后的修改

1. 返回 Step 1 时保留当前城市、日期、路线形态和所有偏好。
2. 未修改设置时，“返回行程”不弹确认。
3. 修改任一规划字段后显示：

   ```text
   路线设置已修改。当前行程仍是旧版本，重新生成后才会更新。
   ```

4. 返回旧行程允许继续查看，但不得将旧结果伪装成已同步。
5. 重新生成前继续使用现有设置差异确认路径。
6. 重新生成成功后清除脏状态；失败时保留旧行程与新设置，允许重试。

### FR-13 空、错误与降级状态

| 状态 | 用户反馈 | 是否阻断下一步 |
|---|---|---|
| 路线为空 | “先添加出发地和目的地” | 是 |
| 空输入 | “请输入城市名称” | 否，仅阻断添加 |
| 重复城市 | “该城市已在路线中”并定位原卡片 | 否 |
| 城市名过长/仅标点 | “请输入有效的城市名称” | 否，仅阻断添加 |
| 已有 10 座城市 | “单次最多规划 10 座城市” | 否，仅阻断添加 |
| 全部过境 | “请至少为一座城市安排游玩天数” | 是 |
| 游玩天数超过 30 天 | “单次最多规划 30 个游玩日，请拆分行程” | 是 |
| 过去日期 | “出发日期不能早于今天” | 是 |
| 单城市选择环线 | “添加至少两个城市后可设置环线” | 是 |
| 高德不可用 | “暂时无法确认城市位置，仍可继续规划” | 否 |
| AI 生成失败但有 fallback | 进入 Step 3，明确标注降级来源 | 否 |
| AI 与 fallback 均失败 | 停留 Step 2，保留全部设置 | 是，针对生成 |

### FR-14 焦点、键盘与无障碍

1. 所有输入、按钮、复选框、选择器均有可见 `focus-visible`。
2. 分段控件使用 `radiogroup`/`radio` 语义，并同步 `aria-checked`。
3. 锁定步骤同时设置 `disabled` 或 `aria-disabled`，并给出原因。
4. 动态错误使用 `aria-live="polite"`；删除撤销使用 `role="status"`。
5. 卡片排序按钮包含城市名和结果，例如“将合肥上移到第 1 站”。
6. 拖拽不是唯一排序方式。
7. 减少动态偏好开启时关闭卡片入场和大幅位移动画。
8. 图标按钮触控目标不小于 44×44 CSS 像素。

### FR-15 响应式规则

#### 桌面（>900px）

- 保持自然页面滚动；
- 路线列表使用当前卡片语言；
- 提供粘性摘要或粘性操作区；
- 地图仍按需打开，不占用 Step 1 常驻主区域。

#### 窄屏（≤900px）

- 步骤条横向展示；
- 卡片控制允许换行，标签不得被截断到无法理解；
- 上移、下移、删除始终可见；
- 不依赖 HTML5 拖拽；
- 底部操作条考虑安全区 `env(safe-area-inset-bottom)`；
- 不产生横向页面滚动。

## 11. 数据与接口要求

### 11.1 `/api/plan` 请求保持不变

```json
{
  "destinations": [
    {
      "name": "淮北",
      "days": 0,
      "plan_stay": false,
      "transport": "auto"
    },
    {
      "name": "合肥",
      "days": 1,
      "plan_stay": true,
      "transport": "driving"
    }
  ],
  "days": 1,
  "start_date": "2026-07-20",
  "route_shape": "round_trip",
  "global_transport": "auto"
}
```

约束：

- `destinations` 顺序不可被前端或后端静默重排；
- 第一城 `transport` 不参与到达段；
- `days` 继续等于游玩城市天数之和；
- 过境城市保留在 `destinations`；
- 外部数据与用户城市名继续作为不可信数据处理，插入 HTML 前必须转义。

### 11.2 快照兼容

- 继续兼容缺少 `plan_stay` 的旧快照；
- 旧数据按当前规则用 `days>0` 推断游玩状态；
- 第一城 `days=0` 时推断为过境起点；
- 不批量重写 localStorage；
- 本轮不提升快照 schema 版本。

## 15. 手工验收用例

### AC-01 正常环线

1. 添加淮北、合肥、武汉；
2. 淮北仅过境，合肥 1 天，武汉 2 天；
3. 选择环线；
4. 合肥到达方式选自驾，武汉选高铁；
5. 观察摘要并进入 Step 2。

预期：

- 摘要为“淮北 → 合肥 → 武汉 → 淮北”；
- 共 3 个游玩日、2 个游玩城市、1 个过境城市；
- 合肥显示“淮北 → 合肥 · 自驾优先”；
- 武汉显示“合肥 → 武汉 · 高铁优先”；
- Step 2 可进入。

### AC-02 排序改变起点

1. 在 AC-01 中把武汉移到第一位；
2. 检查卡片、摘要和提示。

预期：

- 武汉变为出发地；
- 环线返回武汉；
- 所有“上一段”文案重算；
- 页面提示检查段级交通；
- 天数和游玩状态不变。

### AC-03 无效输入

依次尝试空输入、重复城市、纯标点和超长名称。

预期：每种操作都有就近提示；已有路线不被破坏；重复城市卡被定位。

### AC-04 全程过境

把所有城市切换为仅途经。

预期：下一步禁用并显示明确原因；不进入 Step 2。

### AC-05 删除与撤销

删除中间城市后点击撤销。

预期：城市恢复到原位置，天数、游玩状态和交通偏好完整恢复。

### AC-06 日期

选择昨天、清空日期、选择未来日期。

预期：昨天阻断；清空显示“日期未定”但允许继续；未来日期正常。

### AC-07 已生成结果后修改

生成一份行程后返回 Step 1 改变顺序。

预期：旧行程保留，页面显示待重新生成；生成失败时仍可返回旧行程。

### AC-08 窄屏和键盘

在 360px 宽度下只用键盘完成添加、排序、改天数、选择环线和下一步。

预期：无横向滚动；焦点可见；所有功能可达；底部操作不遮挡内容。

## 18. 最终决策

以下事项在本 PRD 中已锁定，实施时无需再次产品确认：

1. 保留当前视觉语言和三步向导，不推翻页面结构。
2. 第一城继续作为出发地，不新增独立 origin 字段。
3. 到达方式继续保存在当前城市，语义固定为“上一城 → 当前城”。
4. 过境城市保留在路线中，但不生成游玩日。
5. 总数统一称为“游玩天数”，不隐式增加转场日。
6. 单城市不能选择环线。
7. 删除采用即时删除 + 5 秒撤销，不使用阻断确认框。
8. 日期可以为空，但过去日期不允许继续。
9. 修改后的设置不会静默覆盖旧行程，必须重新生成。
10. P0 不依赖新的在线城市搜索或付费服务。
