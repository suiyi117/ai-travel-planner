# AeroTravel 30 天验证路线图

> 状态：Current  
> 周期：首轮连续 30 天  
> 更新日期：2026-07-15  
> 上位文档：[产品总纲](产品总纲.md)

## 1. 本轮目标

本轮只验证一个核心命题：

> 是否有人愿意为非标准的多城市、多景点联动规划付费，并且 AeroTravel 能否帮助经营者以可持续的时间成本完成交付。

目标值：

- 发布 6–10 篇目的地组合内容。
- 获得至少 5 次有效咨询。
- 完成至少 1 次真实付费。
- 完成至少 3 次完整交付演练或真实交付。
- 记录每次从需求确认到发布的耗时与返工原因。

“有效咨询”指用户愿意提供目的地、日期或天数、预算、同行人与偏好等具体信息，不包括单纯点赞、收藏和泛泛询问。

## 2. 执行原则

- 本轮默认功能冻结，只完成已经在途、直接影响稳定交付的工作。
- 遇到问题优先用人工流程解决并记录，不立即开发自动化。
- 每项新增开发必须对应真实订单阻塞或重复出现的交付问题。
- 每周只评审一次需求池，避免被当天的新想法持续打断。

## 3. 第 0 阶段：收口当前开发

目标：建立一个可以用于真实演练的稳定基线。

工作：

- 将当前在途修改按“路线设置交互”和“地图/发布页基础设施”分清验收范围。
- 完成自动化检查和浏览器冒烟。
- 确认生成、编辑、地图抽屉、保存、发布专属行程和备份入口均可用。
- 当前批次结束后暂停新的视觉、地图和自驾扩张。

退出条件：

- 当前在途功能有明确完成定义。
- 没有阻塞完整交付演练的高优先级缺陷。
- 可以从一份客户需求走到只读专属行程页。

## 4. 第 1 周：建立可售卖的人工流程

目标：不依赖临场发挥完成一次标准交付。

工作：

1. 固定两种首轮商品：路线会诊、多地联动定制。
2. 写清每种商品的输入、交付内容、交付时限和修改次数。
3. 建立客户需求采集模板，至少包含：
   - 出发地、目的地和必去地点；
   - 日期、总天数与可调整范围；
   - 预算、同行人和旅行节奏；
   - 交通偏好、住宿限制和需要避开的安排；
   - 已购票、已订住宿和其他硬约束。
4. 建立交付前检查清单，至少检查：
   - 城市顺序和交通方向；
   - 每日时间与游玩强度；
   - 地点是否真实、开放信息是否需要复核；
   - 预算和交通信息是否标注为参考；
   - 专属页、导航、文字和 PDF 是否可用；
   - 页面是否包含敏感个人信息。
5. 选择两个有代表性的多地组合，完成端到端演练。
6. 记录每个环节耗时，不在演练中临时扩展产品范围。

周末评审：

- 哪一步最耗时？
- 哪一步最容易出错？
- 哪些输出客户可能根本不需要？
- 现有工作台是否已经足够完成第一单？

## 5. 第 2 周：发布内容并收集咨询

目标：让真实目标用户看到具体问题和具体结果。

工作：

- 发布 2–3 篇目的地组合案例。
- 每篇内容聚焦一个明确难题，例如顺序选择、天数取舍、跨城交通或特殊地点组合。
- 不完整公开付费方案，保留咨询理由。
- 使用统一需求采集模板回复咨询。
- 记录用户原话，不把解决方案猜测当作用户需求。

本周不因单个评论开发功能。只有影响当前交付的故障允许插入修复。

## 6. 第 3 周：完成首批真实或低价交付

目标：验证购买意愿和实际使用价值。

工作：

- 继续发布 2–3 篇案例。
- 优先推动有效咨询进入路线会诊或完整定制。
- 按标准流程完成交付。
- 交付后询问：
  - 哪一部分最有用？
  - 是否真的打开过专属页、地图、导航、文字或 PDF？
  - 哪些安排需要修改？
  - 如果没有购买，主要阻力是什么？
- 记录制作耗时、修改次数和发布耗时。

## 7. 第 4 周：复盘并决定下一阶段

目标：根据证据选择下一条产品路线。

完成剩余内容发布，并汇总漏斗：

```text
内容曝光
  → 有效咨询
  → 明确报价
  → 付费
  → 完成交付
  → 确认有用
```

按结果决策：

### 有咨询、有付费，但制作太慢

优先优化工作台、模板复用、质量检查和发布自动化。

### 有咨询、有付费，客户频繁要求修改或更新链接

评估可更新的分享快照、访问码、有效期和轻量服务端存储。

### 大量用户明确要求自己在线生成

再评估自助 SaaS、账号、云端保存和支付，不提前建设。

### 有浏览和互动，但没有有效咨询

优先检查内容选题、咨询入口和价值表达，不继续增加产品功能。

### 有咨询但无人付费

优先检查目标用户、商品边界、样例可信度、价格和交付承诺。

## 8. 每单记录表

每次演练或真实订单至少记录：

| 字段 | 说明 |
|---|---|
| 来源内容 | 用户从哪篇内容或哪个渠道进入 |
| 需求类型 | 路线会诊 / 多地联动定制 |
| 地点组合 | 城市与核心地点 |
| 是否有效咨询 | 是否提供了足够具体的需求 |
| 是否报价 | 是 / 否 |
| 是否成交 | 是 / 否及原因 |
| 规划耗时 | 需求确认到草案完成 |
| 校对耗时 | 人工检查与调整 |
| 发布耗时 | 成稿到交付链接可用 |
| 修改次数 | 交付前后修改次数 |
| 使用内容 | 专属页 / 导航 / 文字 / PDF |
| 主要问题 | 错误、返工或用户反馈原话 |

第一阶段使用本地表格即可，不为记录系统开发新功能。

## 9. 本轮完成定义

- 达到内容发布、有效咨询和付费目标，或获得足以否定当前假设的明确证据。
- 至少完成 3 次可复盘的完整交付流程。
- 能说清单次交付的主要耗时和错误来源。
- 能用客户证据决定下阶段是优化生产、升级交付、探索自助化，还是调整获客与定位。
- 下一阶段需求来自本轮证据，而不是来自新的界面灵感。
