## 1. 状态定义

| 状态 | 含义 |
|---|---|
| Active | 已进入当前执行周期，有明确验收标准 |
| Next | 当前周期完成后最可能进入，但仍需证据确认 |
| Candidate | 有价值假设，尚无足够证据，不排期 |
| Triggered only | 只有出现列出的真实业务信号后才评估 |
| Done | 已满足当前阶段要求，仅做维护 |
| Superseded | 已被当前产品方向或后续方案替代，不再指导开发 |
| Historical | 决策或实施记录，仅供追溯 |

未经本文件改为 Active 的 PRD，不代表已经排期。

## 5. 当前冻结项

以下方向在 30 天验证期内不新增范围：

- 新地图模式、地图主题和非必要视觉细节。
- 额外动效、装饰性组件和设计系统扩张。
- 自驾算法扩张及与真实订单无关的边界覆盖。
- 新的导出格式。
- 新的 AI 模式或仅为模型能力展示的功能。
- 前端框架迁移和非必要架构重写。
- 账号、数据库、支付和公开运营系统。

允许修复阻塞交付的缺陷，但缺陷修复不得顺带扩展产品范围。

## 7. 需求进入 Active 的模板

新需求只有补齐以下内容后才能改为 Active：

- 用户问题：谁在什么场景遇到了什么问题？
- 证据：来自订单、访谈、交付耗时还是重复错误？
- 目标指标：希望提高或降低什么？
- 最小范围：最小可验证方案是什么？
- 非目标：这次明确不解决什么？
- 验收标准：怎样判断已经解决？
- 退出或回滚：效果不成立时如何停止？

## 8. 每周评审规则

每周固定评审一次，不随时插单：

1. 查看有效咨询、成交、交付耗时、修改次数和错误记录。
2. 判断 Active 项是否仍直接服务当前验证目标。
3. 只允许一个主要产品 PRD 处于 Active。
4. 新想法先进入 Candidate，并写清触发条件。
5. 已完成计划转为 Done 或 Historical，不继续以“顺手优化”为理由延长范围。
