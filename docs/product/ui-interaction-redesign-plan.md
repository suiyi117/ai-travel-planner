# AeroTravel 前端 UI 与交互重构计划

> 版本：v2 · 2026-07-14  
> 状态：待评审  
> 范围：仅前端，不改后端  
> 视觉方向：Claude / Anthropic 风格的羊皮纸、赤陶、编辑感与克制留白

## 1. 意图摘要

当前页面已经具备三步向导、城市拖拽、地图抽屉、行程编辑与交付能力，但整体体验仍像“功能陆续叠加到同一张页面”——层级不够清楚、结果页过于拥挤、部分新增控件甚至退化成浏览器默认样式。

本轮不做零散换色，而是把 AeroTravel 收敛成一条清晰的产品路径：

`定路线 → 设偏好 → 浏览行程 → 进入编辑 → 查看洞察 / 地图 → 交付`

完成标准：

- 用户在生成结果后的 10 秒内能看懂“今天去哪、下一步做什么”。
- 每个页面状态只有一个主动作，次要动作被收纳而不是平铺争抢。
- 城市与行程节点可拖拽，并始终保留键盘 / 按钮操作作为可访问兜底。
- 桌面端有分层与留白，移动端是重新编排后的布局，不是缩小版桌面页。
- 结果页所有折叠、筛选、编辑、地图、导出状态都有完整视觉样式与反馈。
- `.\scripts\check.ps1`、前端测试、浏览器冒烟通过；工作区不产生后端文件 diff。

## 2. 已知基线与问题诊断

### 2.1 截图直接暴露的问题

附件 `image.png` 中“城际交通”和“费用与贴士”显示为灰色原生按钮。这不是审美偏好，而是样式契约缺失：`static/index.html` 已有 `.fold-section`、`.fold-trigger`、`.fold-body` 结构，但 `static/styles.css` 没有对应的组件样式，导致浏览器默认按钮、默认字重和默认边框直接露出。

必须先修复这类“结构已上线、样式未接上”的回归，再进入精细视觉调整。

### 2.2 当前产品层面的主要问题

| 问题 | 影响 | 重构方向 |
|---|---|---|
| Step 3 仍是长堆叠 | 结果、编辑、费用、贴士、交通互相抢焦点 | Browse / Edit / Insights 三层模式 |
| 结果工具条权重接近 | 看地图、刷新、重新生成、编辑难以排序 | 保留一个主动作，其余收纳为次级工具 |
| 折叠模块样式缺失 | 页面出现原生按钮，破坏高级感与可理解性 | 建立统一 Fold 组件样式、状态与动效 |
| Day / Filter / CTA accent 过量 | 赤陶失去“高信号”的意义 | accent 只留给主 CTA 与必要 active 状态 |
| Step 1 控件密度仍高 | 多城市、天数、交通、日期集中后容易疲劳 | 按“路线顺序 / 每城停留 / 出发信息”分区 |
| 导出入口存在重复 markup | 桌面与移动端文案容易漂移 | 共享同一份菜单数据与渲染逻辑 |
| 地图、详情、行程关系不够明确 | 点击卡片后用户不知道上下文是否切换 | 地图抽屉绑定当前节点并恢复焦点 |
| 现有 critique.json 仅代表 city-card | 不能证明整页达到 4 分以上 | 建立整页五轴复评，而不是沿用局部评分 |

### 2.3 工作区约束

- 保持 FastAPI + vanilla static frontend，不引入 React、Vue、构建步骤或数据库。
- 只修改 `static/*` 与必要的 `tests/frontend/*`、文档；不改 Python、API 合约和后端服务。
- 保留单一可变 `state`、`applyPlan()`、`localStorage` 行程快照和现有 `/api` 调用方式。
- `state.cities[].days` 仍是城市天数唯一真相，总天数继续自动派生。
- 保留现有 A+B 克制动效：向导切换、紧凑摘要、城市新增卡片、步骤编号；不增加全页装饰动画。
- 现有工作区有未提交的前端改动，且当前分支显示为 `master`。实现前不得自动清理、回滚或重置这些改动。

## 3. 目标用户与关键任务

| 用户 | 主要任务 | 成功时刻 |
|---|---|---|
| 个人旅行规划者 | 添加多城市、安排天数、生成并微调行程 | 不看说明也能完成一条路线 |
| 需要交付行程的人 | 快速改点、调整顺序、导出客户版本 | 修改后知道哪些内容已应用、如何交付 |

## 4. 目标信息架构

保持 `static/index.html` 作为单 SPA 壳，不拆成多个 HTML；用明确的模式和抽屉形成多层次体验。

```text
App Shell
├── Topbar：品牌 · 当前路线摘要 · 我的行程 · 交付
├── Wizard Rail：路线 / 偏好 / 行程
└── Main
    ├── Step 1 路线
    │   ├── 添加城市
    │   ├── 城市顺序与停留天数
    │   └── 出发日期与总天数摘要
    ├── Step 2 偏好
    │   ├── 旅行节奏
    │   ├── 城际交通 / 自驾偏好
    │   ├── 预算
    │   └── 补充要求
    └── Step 3 行程
        ├── Browse：默认主路径，日程时间线
        ├── Edit：编辑台、拖拽、草稿、优化 diff
        ├── Insights：费用、贴士、质量信息
        ├── Map Drawer：地图与当前地点详情
        └── Delivery Menu：客户交付、备份、检查
```

### 4.1 Step 3 默认 Browse

生成完成后默认只呈现：

1. 行程标题与一句解释。
2. 天数 / 城市 / 节点三个紧凑指标。
3. 一个明确的“编辑行程”入口，以及看地图、刷新班次、重新生成等次级操作。
4. 日期导航、类型筛选、当天时间线。
5. 默认收起的城际交通与费用 / 贴士模块。

筛选“交通”时，城际交通模块必须自动展开并卡片化；切回其他筛选时恢复为收起状态，避免结果页重新变长。

### 4.2 Edit 模式

- 入口只保留一个主入口：Browse 工具区的“编辑行程”。
- 左侧为城市顺序和想去清单，右侧为日期与节点列表；窄屏上下叠放。
- 城市和节点都显示明确拖拽把手、抓取态、放置目标和完成反馈。
- 同日重排直接拖拽；跨日使用“移到其他日期”完成，避免跨日拖拽产生隐性错误。
- 底部草稿操作条只在 Edit 模式出现。
- 智能优化成功后，`#candidateDiffPanel` 默认展开；用户可以手动收起、应用或放弃。

### 4.3 Insights 模式

费用、出行贴士和质量检查不与时间线同时争夺首屏。默认收起，打开后使用同一套折叠容器和明确标题，不再出现原生按钮样式或无层级的大块卡片。

## 5. 统一交互规则

### 5.1 控件与状态

- 每个面板只允许一个 terracotta 实心主 CTA；其他按钮使用 warm-sand、surface 或 ghost。
- 任何 `button` 都必须有专属 class、hover、focus-visible、active、disabled 状态，禁止依赖浏览器默认外观。
- 所有折叠按钮必须同时更新 `aria-expanded`、`hidden` 和 `data-open`，并显示 chevron 状态。
- loading、empty、error、success 都要有可读文案；不使用只有颜色的状态表达。
- 点击生成、刷新、优化等异步动作后要锁定重复点击，并在完成或失败时恢复焦点 / 状态。

### 5.2 城市与天数

- 输入城市后按 Enter 或点击“添加”都能创建城市卡。
- 城市卡支持拖拽排序；保留上移 / 下移 / 删除按钮作为键盘和触控兜底。
- 每城天数使用步进控件或明确的减 / 数字 / 加结构；不保留“看起来可操作但被 disabled 的 range”。
- 总天数只读显示“合计 N 天”，并明确由城市天数自动汇总。
- 删除城市、改变顺序、切换过境 / 停留后，摘要和校验立即同步。

### 5.3 行程节点拖拽

- 拖拽开始：节点降低透明度，目标区域出现可识别的 drop ring。
- 拖拽经过：只高亮当前可放置位置，避免整列闪烁。
- 放置完成：更新时间线、草稿脏状态、撤销 / 重做栈，并给出短暂状态提示。
- 拖拽不可用时：显示“请先解除固定顺序 / 固定日期”等原因，而不是静默失败。

### 5.4 地图抽屉

- 默认关闭；从结果工具区或行程卡进入。
- 桌面端右侧抽屉，地图与详情为同一上下文；移动端全屏抽屉。
- 当前时间线卡片、地图 marker、详情面板三者保持同一 active 节点。
- 打开时锁定背景滚动；关闭后恢复触发按钮焦点。
- 加载、空态、地图不可用、地点无坐标都有对应说明。

### 5.5 导出与我的行程

桌面与移动端使用同一份菜单数据，分成三组：

```text
客户交付：复制客户行程 / 发布专属行程 / 预览专属页 / 导出每日图 / 导出总览图
备份：导出 PDF / 导出日历
检查：参考链接检查
```

菜单打开、关闭、点击外部区域、按 Escape 都要有稳定行为；“我的行程”空态、读取失败、删除确认都要有明确反馈。

### 5.6 动效

- 向导切换使用 8px 上移 + fade，约 200ms。
- 紧凑摘要和新城市卡只做一次入场。
- 折叠模块使用短促高度 / opacity 过渡，但不能阻塞阅读。
- `prefers-reduced-motion: reduce` 下关闭非必要动画。
- 不添加无限循环、背景漂浮、逐卡大范围 stagger。

## 6. 视觉系统收敛

使用项目现有 Claude tokens，不新增独立主题。

| 角色 | 规则 |
|---|---|
| 背景 | `var(--bg)` 羊皮纸；主内容使用 `var(--surface)` |
| 主文字 | `var(--fg)`；辅助说明使用 `var(--muted)` / `var(--meta)` |
| 主 CTA | `var(--accent)`，每个视图最多一个主要实心按钮 |
| 边界 | `var(--border)` / `var(--border-soft)`，优先 ring 而不是厚阴影 |
| 标题 | `var(--font-display)`，功能性文字使用 `var(--font-body)` |
| 圆角 | 普通控件 `var(--radius-sm/md)`；大容器 `var(--radius-lg)` |
| hover | 从 surface 转向 surface-warm 或 ring，不使用裸 `#fff` |

视觉硬规则：

- 不使用紫色渐变、冷灰、默认浏览器按钮、重 drop-shadow。
- 不把每个标题都配一个图标，不用卡片堆叠代替信息层级。
- 不把 accent 同时铺到 day tab、filter tab、时间线节点、多个 CTA。
- 结果页首先像一份可读的行程单，其次才像一个工具面板。

## 7. 分阶段实施任务

### Phase 0：基线与样式契约

**目标：** 先解决“结构存在但样式缺失”的高风险回归。

- [ ] 为 `.fold-section`、`.fold-trigger`、`.fold-trigger-main`、`.fold-meta`、`.fold-chevron`、`.fold-body` 建立完整组件样式。
- [ ] 统一 `data-open="true|false"`、`aria-expanded`、`hidden` 的状态映射。
- [ ] 盘点 `index.html` 中新增 class 与 `styles.css` 的 selector 覆盖，补齐结果层、洞察层、交通段卡片的缺口。
- [ ] 清理结果区不必要的 inline layout style，避免响应式规则被 HTML 内联样式锁死。

验收：截图中不再出现原生按钮；折叠条、展开体、chevron、hover、focus-visible 在 1440px / 390px 都有完整表现。

### Phase 1：App Shell 与 Step 1 路线

**目标：** 让路线输入成为最短、最可信的第一步。

- [ ] 把 Step 1 重排为“城市序列 → 每城停留 → 出发信息”三个视觉区。
- [ ] 统一城市卡的标题、拖拽把手、过境状态、天数步进、交通选择和操作按钮。
- [ ] 让添加城市、删除城市、排序、切换停留状态即时同步摘要与校验。
- [ ] 把日期和自动汇总天数放到同一信息块，避免用户误以为总天数可以单独修改。
- [ ] 补齐空城市、重复城市、日期缺失、城市删除后的错误与恢复路径。

依赖：Phase 0。文件：`static/index.html`、`static/styles.css`、`static/app.js`、`static/wizard.js`、对应前端测试。

### Phase 2：Step 2 偏好

**目标：** 减少表单疲劳，保证生成动作明确。

- [ ] 将节奏、交通、自驾偏好、预算、补充要求分成清晰的分组，而不是连续堆叠。

- [ ] 生成中锁定按钮、显示可理解的进度状态；失败时保留用户填写内容并提供重试。

### Phase 3：Step 3 Browse

**目标：** 让结果页默认只服务于“浏览今天的行程”。

- [ ] Day tabs 与 Filter tabs 使用 surface + ring active，不使用整块 accent 填充。
- [ ] 时间线卡统一 hover、active、交通、景点、美食、住宿信息层级。

### Phase 4：Edit 与拖拽闭环

## 9. 风险与处理

| 风险 | 影响 | 处理 |
