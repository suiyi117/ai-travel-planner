# AeroTravel 前端 UI 与交互重构计划

> 版本：v3（整理版） · 2026-07-15
> 状态：已确认首轮范围，待实施
> 适用范围：仅前端；后续实现只修改 `static/index.html`
> 视觉方向：Claude / Anthropic，羊皮纸底色、赤陶主强调、文学沙龙感

## 0. 当前结论

这份文档合并此前多轮追加的 UI 调整，作为后续实现的单一计划来源。重点不是继续堆叠卡片或信息区，而是把已有信息重新编排成更完整的阅读节奏：

1. 顶部 App Shell 保持稳定，品牌、路线摘要和操作区始终可见。
2. 结果页默认只展示浏览主路径，编辑、洞察和费用信息按需进入。
3. 时间从行程卡片内部移到左侧时间线节点下方，成为清晰的节奏标记。
4. 行程卡片用“内容主体 + 轻量元信息”补足构图，但只使用已有数据，不虚构指标。
5. “城际交通”和“费用与贴士”的折叠入口改为与羊皮纸 / 赤陶视觉体系一致的自定义控件。
6. 地图继续保持宽幅主画布，详情使用浮层或移动端底部页片，不恢复固定左右分栏。

本轮只整理计划文件，不修改页面代码、样式文件、脚本或后端。

本轮确认：Insights 保留折叠区；行程卡片采用右侧轻量元信息轨；顶部先维持路线摘要与操作稳定，移动端细节在冒烟检查后微调；首轮实施范围锁定为 Phase 0–3。

## 1. 意图与成功标准

### 1.1 产品意图

把 AeroTravel 从“信息都能看到”整理为“用户知道下一步该看什么、该做什么”的行程规划工作台：

> 定路线 → 设偏好 → 浏览行程 → 进入编辑 → 查看洞察 / 地图 → 交付

### 1.2 成功标准

- 用户进入结果页后约 10 秒内能回答：今天去哪、下一站是什么、什么时候出发。
- 每个状态只有一个视觉主动作，不让生成、编辑、地图、导出互相竞争。
- 时间节点在时间线上独立可扫读，不再埋在卡片标题区域。
- 卡片不出现大片无意义留白，也不通过虚构评分、效率、节省金额等数字填充。
- 桌面端有清晰的纵向节奏和足够留白；移动端是重新编排后的单列，而不是压缩桌面布局。
- 所有折叠、筛选、编辑、拖拽、地图抽屉和导出入口都有明确的 hover、focus、active、loading、empty、error、success 状态。
- 前端回归测试和浏览器冒烟检查通过；后端接口契约不变。

## 2. 已确认的产品与工程边界

### 2.1 修改范围

- 后续实现只允许修改 `static/index.html`。
- 不修改 `static/styles.css`、`static/app.js`、其他前端模块、后端路由、数据结构或 API 契约。
- 不引入 React、Vue、构建步骤、数据库、登录系统或新的付费 API。
- 不添加产品中不存在的设计设置、视口切换器、平台选择器、主题旋钮、演示控制面板。

### 2.2 实施流程

- 进入代码实施前，先确认工作区状态并创建 / 切换到专用 feature 分支。
- 保留现有未提交改动，不使用破坏性回滚覆盖用户工作。
- 每个阶段完成后只做与本计划相关的前端检查。
- 若某项必须修改计划外文件，应先暂停并征得确认。

### 2.3 已有实现作为基线

- 当前页面已经有滚动式三步向导、顶部上下文区域、宽幅地图抽屉和结果页三层信息架构的基础。
- 当前动效范围锁定为 A+B：向导面板切换、紧凑摘要 / 新城市卡片的克制入场，以及步骤点的轻微反馈；不扩展为全页面装饰动画。
- 当前结果页默认采用 Browse / Edit / Insights 的分层结构；跨天移动继续通过“移到其他日期”，不引入自由跨天拖拽。

## 3. 当前基线与问题诊断

### 3.1 行程卡片显空

参考截图：`drawing-2026-07-15T04-32-02-588Z.png`。

当前卡片的内容主要集中在左上区域，右侧和下方留白较大。问题不在于信息不足，而在于卡片缺少“时间线关系”和“元信息收束”：

- 时间仍在卡片内部，卡片承担了时间、标题、描述和操作四种层级，视觉重心不明确。
- 现有类型、城市、时长、交通方式等信息没有形成轻量摘要层。
- 若直接增加更多文字，容易重新变成拥挤的说明卡；若增加新指标，则会造成数据虚构。

### 3.2 折叠入口不属于当前视觉体系

参考截图：`drawing-2026-07-15T04-29-46-560Z.png`。

`#transportFoldBtn` 与 `#insightsFoldBtn` 当前使用 `.fold-section` / `.fold-trigger` 结构，但缺少对应的目标样式，浏览器默认按钮外观被暴露出来。需要保留现有 DOM、ID、ARIA 和状态逻辑，只在 `static/index.html` 中补充目标范围内的样式。

### 3.3 时间层级不够醒目

参考截图：`drawing-2026-07-15T04-31-23-478Z.png`。

`.item-time` 当前位于 `.itinerary-card` 内部。目标结构是：

> 时间线节点 → 节点下方的时间 → 行程卡片

时间应成为用户扫读行程节奏时的独立锚点，同时保留现有时间值、点击逻辑、地图联动、active / conflict 状态和移动端行为。

### 3.4 顶部上下文需要保持稳定

品牌、路线摘要、步骤状态和操作区不能因步骤切换、滚动或状态卡移动而消失或跳动。状态提示可以放到左侧步骤区域下方，但不能以此牺牲顶部核心上下文。

## 4. 目标信息架构与主路径

### 4.1 App Shell

稳定存在的全局层：

- 品牌与产品名称。
- 当前路线摘要：出发地、目的地、日期 / 天数等已有信息。
- 当前步骤状态与主要操作。
- 仅在确有需要时出现的辅助状态提示。

桌面端使用稳定顶部区域；移动端压缩间距和信息密度，但不隐藏当前路线和主要操作。移动端隐藏大块 `wizard-summary`，仅保留必要的步骤点与上下文。

### 4.2 Step 1：路线

- 出发地与多城市目的地输入。
- 每个城市独立的天数设置，保持 `state.cities[].days` 为单一事实来源。
- 城市新增、删除、排序和校验反馈。
- 可进入下一步的明确主操作。

### 4.3 Step 2：偏好

- 旅行节奏、预算、兴趣和交通偏好。
- 输入错误、缺少必要条件和生成中的状态。
- 生成操作锁定，避免重复点击；完成后进入 Step 3。

### 4.4 Step 3：结果页三层结构

#### Browse：默认浏览路径

- 默认只展示按日期组织的时间线。
- 每日摘要、时间节点、行程卡片和必要的交通节点作为主内容。
- 城际交通默认折叠；当筛选条件为 `transport` 时强制展开并卡片化展示。
- 费用与贴士 / Insights 默认折叠或按需进入，不与时间线并排争夺首屏注意力。

#### Edit：编辑台

- 城市和同日节点可拖拽。
- 跨天移动使用“移到其他日期”的明确动作，不用自由拖拽制造隐含规则。
- 编辑态显示草稿操作栏；浏览态不显示编辑专用控件。
- “智能优化”成功后，diff 对比面板默认展开。

#### Insights：洞察与交付

- 费用、贴士、住宿、美食和交付操作按需展开。
- 导出、分享、复制等动作合并为一个清晰的交付入口，避免重复操作组。

### 4.5 地图抽屉

- 地图始终是宽幅主画布，不使用常驻的固定左右详情栏。
- 景点详情以地图浮层、节点条或底部页片呈现。
- 移动端使用接近全屏的地图和底部详情页片。
- 地图关闭后回到原时间线位置，不打断当前浏览上下文。

## 5. 已确认的重点改造

### 5.1 顶部上下文与左侧状态提示

#### 目标

建立稳定的 App Shell：顶部保留品牌、路线摘要和操作区；状态提示卡位于左侧步骤导航下方时，成为辅助上下文而不是新的主面板。

#### 规则

- 顶部区域的高度和对齐方式在步骤切换前后保持稳定。
- 玻璃摘要使用现有玻璃语言，不新增厚重阴影或独立品牌色。
- 左侧状态卡不重复顶部路线摘要；只呈现当前步骤需要知道的状态。
- 移动端不显示大块摘要卡，只保留步骤点和必要状态。

### 5.2 折叠入口视觉统一

#### 目标选择器

- `.fold-section`
- `.fold-trigger`
- `#transportFoldBtn`
- `#insightsFoldBtn`

#### 视觉方案

- 使用羊皮纸背景上的 Ivory / Warm Sand 表面，配合暖色 ring 边界。
- 标题、说明、当前状态和展开箭头组成一个完整点击区域。
- 入口不使用浏览器默认按钮样式，不使用左侧彩色边框，不添加独立的重阴影。
- 只有展开后的当前内容使用赤陶作为高信号强调；折叠入口本身保持克制。
- 最小点击高度为 44px，键盘 focus 使用统一 focus ring。

#### 交互与状态

- 保留现有 `aria-expanded`、`aria-controls`、`hidden`、`data-open` 等状态同步。
- 运输筛选时强制打开城际交通；离开运输筛选时恢复默认折叠策略。
- 洞察区默认折叠，用户主动打开后保留当前会话中的状态。
- 支持 `prefers-reduced-motion`，折叠动画只做短距离、低幅度的高度 / 淡入过渡。

### 5.3 时间线节点与行程卡片重排

#### 目标构图

桌面端每个行程项采用三段结构：

1. 左侧时间线轨道：节点、连接线、节点下方的醒目时间。
2. 中部内容主体：地点 / 类型、标题、描述、已有操作。
3. 右侧轻量元信息：只组织现有的城市、时长、交通方式、开放时间或其他当前数据。

右侧元信息不是新的详情栏，也不应形成固定地图分栏。它是卡片内部的轻量摘要层，用来平衡构图和帮助扫读。

#### 信息优先级

- 第一层：时间与当前节点状态。
- 第二层：地点 / 类型与行程标题。
- 第三层：描述和已有交互。
- 第四层：时长、城市、交通等辅助元信息。

#### 内容约束

- 只使用现有行程数据；没有真实值时显示 `—` 或隐藏该元信息。
- 不新增“节省 xx 分钟”“效率提升 xx%”“评分 xx”等未经来源支持的数字。
- 不复制顶部摘要、日期标题或卡片内已有信息。
- 不用增加文字量来填空；优先通过网格、对齐、分隔和层级完成构图。

#### 响应式策略

- 桌面端：元信息靠右收束，卡片主体拥有主要宽度。
- 平板端：右侧元信息变为卡片顶部或底部的紧凑横向行。
- 移动端：时间仍位于节点下方；卡片回到单列，元信息以短标签 / 两列小行呈现，不制造横向滚动。
- 不改变时间线节点与卡片的点击、拖拽、地图定位、冲突提示和 active 状态。

## 6. 统一交互规则

### 6.1 控件

- 所有按钮、筛选项、折叠入口、标签和输入控件使用现有暖色视觉语言。
- 不允许浏览器默认按钮、默认 `details/summary` 三角标或孤立的冷色控件出现在结果页。
- 主 CTA 只保留一个赤陶重点；次级动作使用 Warm Sand、Ivory 或文字动作。
- 所有可点击控件达到 44px 触控尺寸，并有清晰的 hover / focus / active / disabled 状态。

### 6.2 状态

每个主要模块需要覆盖：

- 初始 / 默认。
- 加载中。
- 空状态。
- 错误与可恢复操作。
- 成功 / 已保存。
- 禁用或权限不足（如适用）。

状态文案应告诉用户发生了什么以及下一步可以做什么，避免只显示原始错误字符串。

### 6.3 拖拽与键盘替代

- 同日拖拽时显示明确的插入位置和目标反馈。
- 跨天移动通过“移到其他日期”完成，避免依赖拖拽完成复杂规则。
- 键盘用户可以通过聚焦、移动按钮或日期选择动作完成同等操作。
- 拖拽过程中防止重复提交和异步状态覆盖；完成后保留原有焦点上下文。

### 6.4 动效

- 向导面板切换：约 200ms，淡入并上移约 8px。
- 紧凑摘要与新城市卡片：只在首次进入时做一次克制入场。
- 步骤点：轻微缩放反馈，约 150ms。
- 不对所有卡片做级联动画，不使用持续漂浮、呼吸或装饰性循环动画。
- `prefers-reduced-motion: reduce` 时关闭位移和非必要过渡。
- 处理点击防抖，避免快速连续点击生成、折叠或导出。

## 7. 视觉系统

### 7.1 颜色与表面

使用项目已绑定的 Claude / Anthropic 设计系统：

- 页面底色：`--bg`，羊皮纸感。
- 卡片表面：`--surface`，用于主要内容容器。
- 交互表面：`--surface-warm`，用于次级按钮、折叠入口和弱强调区域。
- 主文字：`--fg`；辅助文字：`--muted`；元信息：`--meta`。
- 边界：`--border` / `--border-soft`。
- 主强调：`--accent`，仅用于主 CTA、关键节点和最高信号状态。

禁止在本计划范围内引入新的色彩体系、冷蓝灰大面积背景、紫色渐变、粉色 / 橙棕泛滥或纯白页面底色。

### 7.2 字体与层级

- 标题使用 `--font-display`，保持文学感和稳定的中等字重。
- UI、按钮、标签和描述使用 `--font-body`。
- 时间、编号和小型数据使用 `--font-mono`，只在确有数据属性时使用。
- 时间必须比卡片内的辅助元信息更醒目，但不能抢过日期标题和当前主动作。
- 正文保持舒适行高；通过层级、间距和对齐改善密度，不靠缩小文字填充空间。

### 7.3 深度与形状

- 优先使用 ring 边界和轻微 elevation，不使用厚重 drop-shadow。
- 普通卡片使用克制圆角；主要容器可使用更大的圆角，但不能让每个子元素都变成胶囊。
- 一张卡只保留一个主要视觉焦点，避免每个标题旁都放图标或彩色徽标。

## 8. 响应式与可访问性

### 8.1 断点策略

- 360 / 390 / 430px：重点验证时间线、卡片和底部操作不横向溢出。
- 600–744px：折叠入口和元信息进入紧凑单列 / 横向行。
- 768–834px：验证平板纵向阅读节奏和触控空间。
- 1024px 以上：恢复桌面端的时间线 + 主体 + 轻量元信息构图。
- 1366 / 1440 / 1920px：验证顶部稳定、地图宽幅和主列不过度拉伸。

### 8.2 关键检查

- 顶部品牌、路线摘要和操作区在滚动、步骤切换后仍存在且不跳动。
- 移动端隐藏大块 `wizard-summary`，不压缩主内容的纵向空间。
- 时间节点在所有尺寸下都能与对应卡片明确对应。
- 折叠入口文字、箭头和状态在小屏不挤压、不换行破坏点击区域。
- 键盘 Tab 顺序符合视觉顺序；focus 不被 `overflow` 或动画裁切。
- 颜色对比度、表单标签、按钮名称、折叠 ARIA 和地图关闭动作可被辅助技术理解。

## 9. 分阶段实施

### Phase 0：整理基础控件

- 在 `static/index.html` 中为 `.fold-section`、`.fold-trigger`、`#transportFoldBtn`、`#insightsFoldBtn` 补充目标范围内的样式。
- 复核按钮默认样式、focus、disabled、展开 / 折叠状态和 reduced motion。
- 不改变折叠逻辑、ID、ARIA 或数据属性。

### Phase 1：稳定 App Shell 与 Step 1

- 固定顶部品牌、路线摘要和操作区的结构节奏。
- 整理左侧步骤导航下方的状态提示，去掉与顶部摘要重复的信息。
- 复核多城市输入、天数节奏、城市新增卡片和移动端布局。

### Phase 2：Step 2 偏好与生成状态

- 统一偏好控件、校验、生成中、成功和失败状态。
- 增加点击防抖和可恢复错误提示的视觉反馈。
- 保持现有生成接口和状态对象不变。

### Phase 3：Step 3 Browse 结果页

- 先完成时间线节点与 `.item-time` 的视觉重排。
- 再完成行程卡片的“主体 + 轻量元信息”构图，优先使用现有数据。
- 接入统一折叠入口的视觉样式和运输筛选强制展开规则。
- 保持 Browse 默认只露出时间线主路径。

### Phase 4：Edit 编辑闭环

- 保留同日节点拖拽和城市调整。
- 明确跨天“移到其他日期”的反馈与撤销 / 保存状态。
- 确保智能优化成功后 diff 默认展开，草稿栏只在编辑态出现。

### Phase 5：地图与交付

- 保持宽幅地图主画布。
- 复核浮层、节点条、移动端底部页片和关闭回到原位置的行为。
- 合并导出、分享、复制等交付动作，减少重复入口。

### Phase 6：移动端与动效收口

- 逐一检查 360 / 390 / 430 / 768 / 1024px 等关键宽度。
- 验证顶部上下文、时间节点、折叠入口和底部操作的触控可用性。
- 完成 A+B 范围内的动效与 reduced-motion 收口。

### Phase 7：验收与交付
