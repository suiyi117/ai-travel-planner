# AeroTravel 自驾规划交互重构 PRD

> 状态：Proposed，待产品评审
> 日期：2026-07-14
> 范围：三步向导中的交通与自驾设置、结果页编辑入口
> 关联决策：[ADR-002：约束驱动的可编辑规划](../decisions/ADR-002-constraint-driven-editable-planning.md)
> 关联设计：[AeroTravel UI 交互重设计计划](./ui-interaction-redesign-plan.md)

## 1. 背景

当前 Step 1 使用“规划模式”二选一：

- 城市行程
- 自驾路线

该设计把三个不同层级的概念放在同一个控件中：

1. 行程内容组织方式：按天安排景点和活动。
2. 城际交通方式：高铁、飞机、自驾或智能推荐。
3. 生成后的编辑工具：编辑每日安排或优化自驾节点顺序。

“城市行程”和“自驾路线”并不是互斥关系。自驾旅行仍然需要每日行程，城市行程也可能以自驾作为城际交通方式。

当前实现还存在认知与行为不一致：生成前选择 `self_drive` 不会改变 `/api/plan` 的请求结构或 AI 生成逻辑，它主要影响生成后的 `WorkingDraft.mode` 和路线优化能力。因此，“规划模式”会让用户误以为自己正在选择两种完全不同的生成结果。

## 2. 目标

本次重构要让用户按照真实任务顺序完成设置：

1. 决定去哪些城市以及是否返回起点。
2. 选择城市之间的主要交通方式。
3. 仅在需要自驾时设置路线偏好。
4. 生成统一的按日行程。
5. 在结果页按需进入每日编辑或自驾路线优化。

完成后，用户不需要理解“规划模式”这一内部概念，也能正确找到自驾能力。

## 3. 成功标准

### 3.1 产品成功标准

- Step 1 不再展示“城市行程 / 自驾路线”二选一。
- 用户可以从“默认城际交通”自然进入自驾设置。
- 选择高铁或飞机时，不展示无关的自驾策略。
- 无论选择何种交通方式，生成结果始终是一份按日行程。
- 自驾用户可以在结果编辑区找到“优化自驾路线”入口。
- 用户切换每日编辑和自驾路线优化时，不丢失已生成行程、草稿修改或路线设置。
- 页面不再声称 AI 会根据 `planningMode` 使用不同生成逻辑。

### 3.2 可用性成功标准

- 首次使用者能够在不阅读长说明的情况下说清楚“路线形态”和“交通方式”的区别。
- 自驾设置只在相关场景出现，默认页面不增加额外认知负担。
- 桌面和移动端使用相同的概念、标签与状态规则。
- 所有单选控件都可通过键盘操作，并由辅助技术读出当前选中状态。

## 4. 非目标

本次不包含：

- 修改 `/api/plan`、`/api/plan/optimize` 或交通接口契约。
- 为 AI Prompt 增加新的“自驾专用生成模式”。
- 新增数据库、账号、云同步或服务端草稿。
- 引入前端框架、状态管理库或构建步骤。
- 重写现有自驾算法、约束校验、路线矩阵或高德降级逻辑。
- 删除草稿 schema 中的 `mode: itinerary | self_drive`。
- 重做整个三步向导或结果页视觉系统。

## 5. 用户与核心场景

### 5.1 普通城市旅行者

用户计划北京、西安、成都的多城旅行，主要乘坐高铁。用户关心每座城市停留几天、景点节奏和城际班次，不需要看到自驾策略。

预期路径：

1. Step 1 添加城市并设置单程或环线。
2. Step 2 选择“高铁”或“智能推荐”。
3. 生成按日行程。
4. 在结果页浏览或编辑每日安排。

### 5.2 自驾旅行者

用户计划成都出发的川西环线，希望减少驾驶时间或兼顾体验，并在生成后调整途经点顺序。

预期路径：

1. Step 1 设置城市、停留天数和“环线”。
2. Step 2 选择“自驾”。
3. 页面展开“自驾偏好”，用户选择少开车、均衡或多体验。
4. 生成统一的按日行程。
5. 在结果页进入“优化自驾路线”，查看道路顺序、里程、时间和变更候选。

### 5.3 混合交通旅行者

用户在长距离城市间乘坐高铁，在其中一段选择自驾。系统需要保留每段交通设置，但不应将整趟旅行强制解释为两套不同的行程。

预期路径：

1. 默认交通选择“智能推荐”或“高铁”。
2. 在城市段设置中将其中一段改为“自驾”。
3. 自驾偏好入口变为可用。
4. 生成结果仍按天组织；自驾优化只处理满足现有能力边界的已定位节点。

## 6. 信息架构调整

### 6.1 Step 1：路线

保留：

- 目的地城市序列
- 每座城市的停留天数和过境状态
- 路线形态：单程 / 环线
- 出发日期
- 自动汇总的计划天数

移除：

- “规划模式”标题
- “城市行程 / 自驾路线”分段控件
- “生成后仍可在结果页切换”的帮助文案

路线形态继续对所有旅行生效：

- 单程：结束于最后一座城市。
- 环线：结束后返回起点，交通段和末日安排体现回程。

### 6.2 Step 2：偏好

“默认城际交通”保留以下选项：

- 智能推荐
- 高铁
- 飞机
- 自驾

当用户明确选择自驾时，渐进展开“自驾偏好”：

- 少开车
- 均衡
- 多体验

建议文案：

> 自驾偏好会影响生成后的路线优化；初始行程仍按天安排景点与活动。

当任一城市段被单独设为自驾时，即使默认交通不是自驾，也应展示一条紧凑提示：

> 当前包含自驾路段，可在生成后优化途经点和道路顺序。

### 6.3 Step 3：结果与编辑

结果页继续以“浏览 / 编辑”作为顶层模式，不新增第三个长期并列的全局模式。

进入编辑后提供两个任务入口：

- 调整每日安排
- 优化自驾路线

“优化自驾路线”只在满足以下任一条件时显示：

- 默认城际交通为自驾。
- 任一城市段交通为自驾。
- 已恢复草稿包含有效的自驾路线数据。

若用户没有选择自驾，不展示禁用按钮。可以在编辑区使用低权重文字入口“需要自驾规划？”引导用户返回交通设置，但不得与主要编辑操作竞争视觉权重。

## 7. 详细功能需求

### FR-01 移除公开的规划模式选择

- Step 1 不再渲染 `#planningModeGroup`。
- 页面不再要求用户在“城市行程”和“自驾路线”之间做互斥选择。
- 现有 `state.planningMode` 可以继续作为内部兼容状态，但不得作为用户必须理解的概念。
- 从旧快照恢复 `planningMode=self_drive` 时，应自动启用自驾编辑能力，不弹出迁移提示。

### FR-02 自驾意图的派生规则

前端统一派生 `hasSelfDriveIntent`，概念规则为：

```text
hasSelfDriveIntent =
  默认城际交通为自驾
  或任一城市段交通为自驾
  或当前草稿已有自驾路线数据
```

约束：

- `智能推荐` 本身不等于明确的自驾意图。
- 派生结果只控制自驾设置和工具的显示，不修改后端 API。
- 用户暂时切换离开自驾时，当前会话可保留最后一次自驾策略；该策略不应影响非自驾优化。

### FR-03 自驾偏好的渐进展示

- 当 `hasSelfDriveIntent=true` 时展示自驾偏好。
- 默认策略为“均衡”。
- 切换策略立即更新前端状态，但不自动发起道路请求。
- 生成前选择策略后，生成完成创建草稿时应写入现有 `strategy` 字段。
- 隐藏自驾偏好时保留本次会话中的选择，避免用户往返交通选项后重复设置。

### FR-04 统一生成语义

- 生成按钮始终使用“生成 AI 旅行规划”或现有等价文案。
- 所有交通选择都生成一份统一的按日行程。
- `/api/plan` 请求继续包含现有城市、天数、交通、预算、兴趣和 `route_shape` 字段。
- 本次不得增加仅由前端消费、后端忽略的伪请求字段。
- 失败时继续使用现有本地演示行程，不因自驾设置改变回退路径。

### FR-05 每日编辑入口

- “调整每日安排”进入现有普通编辑器。
- 支持想去清单、节点移动、约束、撤销、重做和仅保存。
- 打开每日编辑不得清空已有自驾路线数据。
- 每日编辑产生的新地点在具有有效坐标时，可被后续自驾优化使用。

### FR-06 自驾路线优化入口

- “优化自驾路线”进入现有自驾编辑界面。
- 首次进入时使用现有 `ensureSelfDriveDraft()` 或等价路径初始化路线。
- 初始化属于确定性草稿转换，不重新调用 AI。
- 只有用户点击“重新计算道路”或现有等价操作时，才请求驾车路线接口。
- 未定位节点继续显示 warning，不阻断其他已定位节点编辑。
- 少于两个已定位节点时，显示现有可操作错误说明。
- 超过 20 个已定位节点时，继续遵守现有上限并说明原因。

### FR-07 编辑工具之间切换

- 从每日编辑切换到自驾优化时，先提交当前内存中的 `WorkingDraft`，不丢弃未保存修改。
- 从自驾优化切换回每日编辑时，保留路线顺序、路线形态、策略和道路摘要。
- 切换工具不得创建新的 `AppliedPlan`。
- 只有“应用候选方案”或现有保存路径可以更新正式展示结果。
- 若存在未处理的 `CandidatePlan`，切换前应要求用户应用或放弃，避免候选结果失去上下文。

### FR-08 路线形态一致性

- 单程/环线仍在 Step 1 展示，不依赖是否自驾。
- 修改路线形态后，应继续同步行程末日回程和交通段。
- 自驾优化读取相同的 `route_shape`，不得存在第二套单程/环线来源。
- 页面只允许一个可见的路线形态控制源，避免双向漂移。

### FR-09 旧快照兼容

- schema v1 继续只读恢复。
- schema v2 中 `mode=self_drive` 的草稿继续可编辑。
- 旧快照恢复后，自驾入口依据草稿内容自动显示。
- 不批量重写用户的 `localStorage`。
- 本次不提升快照 schema 版本，除非实现阶段发现现有字段无法表达需求；若必须升级，需先更新 ADR。

## 8. 状态规则

### 8.1 用户可见状态

用户只需要理解以下状态：

- 路线形态：单程 / 环线
- 默认交通：智能推荐 / 高铁 / 飞机 / 自驾
- 编辑任务：调整每日安排 / 优化自驾路线

### 8.2 内部兼容状态

现有内部字段继续保留：

- `state.planningMode`
- `WorkingDraft.mode`
- `state.routeShape`
- `state.routeStrategy`

推荐映射：

```text
打开“调整每日安排”
  -> 使用 itinerary 编辑行为

打开“优化自驾路线”
  -> WorkingDraft.mode = self_drive
  -> 初始化或复用 route

退出自驾优化
  -> 不删除 route
  -> UI 返回每日编辑任务
```

实现阶段应避免让 `planningMode` 同时承担“用户交通意图”和“当前编辑面板”两种职责。若需要新增纯 UI 字段，应使用 `editTool: daily | driving` 之类的瞬时状态，且不改变后端 schema。

## 10. 视觉与无障碍要求

- 分段单选继续采用项目现有 parchment、surface 和 ring 视觉系统。
- 选中状态必须同时通过文字、边框或勾选表达，不能只依赖颜色。
- 未选项不得呈现为禁用态。
- 使用原生 `button`，并为单选组提供 `role="radiogroup"`、可识别标签和选中状态。
- 支持 Tab 聚焦、Enter/Space 选择；同一单选组建议支持方向键切换。
- 自驾偏好出现或隐藏时，不抢夺当前输入焦点。
- 渐进区域可使用短过渡，但必须遵守 `prefers-reduced-motion`。
- 320–430px 下控件允许纵向堆叠，不得出现横向滚动。
- 帮助文案使用短句；核心差异不依赖一整段灰色说明文字。

## 11. 异常与边界场景

### 11.1 用户取消自驾

当用户把所有自驾交通改为高铁或飞机：

- 隐藏自驾偏好和自驾优化入口。
- 不立即删除已有路线数据。
- 下次再次选择自驾时恢复本次会话的路线策略。
- 保存非自驾行程时，不让隐藏的自驾策略影响普通行程优化。

### 11.2 混合交通

- 保留每个城市段独立交通选择。
- 自驾优化不能擅自把高铁或飞机段改成自驾。
- 若当前算法只能优化整条自驾路线，应在入口说明能力边界，不伪装成分段道路优化。

### 11.3 外部道路服务失败

- 继续使用现有直线距离估算降级。
- 明确标识估算数据，不阻断草稿编辑和保存。
- 不暴露高德或内部异常文本。

### 11.4 未处理候选方案

- 存在 `CandidatePlan` 时，不允许无提示切换编辑任务。
- 用户可以选择应用候选、放弃候选或留在当前工具。
- 取消操作后保持当前内容和焦点位置。

## 12. 测试策略

### 12.1 前端单元测试

优先新增纯函数测试，避免把状态派生继续堆入 `static/app.js`。

建议覆盖：

- 默认交通为自驾时 `hasSelfDriveIntent=true`。
- 任一城市段为自驾时 `hasSelfDriveIntent=true`。
- 只有智能推荐时不自动判定为自驾。
- 旧自驾草稿恢复后显示自驾入口。
- 切换离开自驾后保留策略，但普通优化不读取该策略。
- 打开自驾优化时初始化路线且不调用 AI。
- 每日编辑与自驾优化切换不丢失草稿 revision 和节点。
- 存在候选方案时阻止无确认切换。
- 路线形态在普通行程和自驾优化中使用同一状态源。

测试文件应放在 `tests/frontend/`，可新增聚焦模块测试，或扩展现有 `draft.test.js`、`draft-ops.test.js` 和 `wizard.test.js`。

### 12.2 浏览器 smoke

桌面宽度至少检查 1024px 和 1440px：

- Step 1 不出现规划模式二选一。
- 选择自驾后自驾偏好正确出现。
- 生成进入 Step 3。
- 每日编辑和自驾路线优化均可进入。
- 地图抽屉、候选 diff 和保存流程不回归。

移动端至少检查 320px、390px 和 768px：

- 分段控件无横向溢出。
- 自驾偏好展开后不遮挡下一步操作。
- 编辑工具入口顺序清晰且可触控。
- 地图抽屉仍可关闭。

键盘检查：

- Tab 顺序符合页面视觉顺序。
- 单选项可通过键盘改变。
- 动态展开区域不会造成焦点丢失。
- 候选方案确认具有明确焦点处理。

### 12.3 质量门

实现完成后执行：

```powershell
.\scripts\check.ps1
```

若修改前端运行时行为，还需按 `docs/smoke-checklist.md` 记录浏览器 smoke 结果。

## 13. 预期影响文件

实现阶段预计涉及：

- `static/index.html`：移除规划模式控件，调整自驾偏好与编辑入口。
- `static/app.js`：状态派生、编辑工具切换和兼容编排。
- `static/state.js`：必要时增加瞬时 `editTool` 状态。
- `static/wizard.js` 或新的聚焦无构建模块：承载纯状态派生规则。
- `static/styles.css`：渐进区域、编辑工具入口和响应式样式。
- `tests/frontend/*.test.js`：状态、兼容与交互回归测试。

默认不修改：

- `server.py`
- `routers/`
- `schemas/`
- `planner/`
- `services/`
- API 文档和环境变量

## 14. 实施顺序建议

1. 抽出并测试 `hasSelfDriveIntent` 等纯状态规则。
2. 移除 Step 1 的规划模式 UI，保留内部兼容字段。
3. 将自驾偏好移动到交通选择之后并实现渐进展示。
4. 在结果编辑区增加每日编辑与自驾优化入口。
5. 实现两个编辑工具之间的无损切换和候选保护。
6. 补齐响应式、键盘与旧快照测试。
7. 执行完整质量门和浏览器 smoke。

## 15. 边界与约束

### 必须做到

- 保持 `state.cities[].days` 为天数来源。
- 保持 `applyPlan()` 为生成和恢复行程的主要 hydration 路径。
- 保持 AppliedPlan、WorkingDraft、CandidatePlan 三阶段模型。
- 保持所有外部数据在 HTML 插入前转义。
- 保持无构建、无框架和 localStorage 持久化。
- 更新相关前端测试并执行完整质量门。

### 实现前需要再次确认

- 是否要把混合交通的自驾优化扩展为严格的“分段路线优化”。
- 是否需要为 `editTool` 持久化；本 PRD 默认它只是瞬时 UI 状态。
- 若现有草稿字段无法无损表达两个编辑工具的切换，是否升级快照 schema。

### 禁止

- 通过新增一个名称不同的全局模式开关重新制造同一问题。
- 为了简化 UI 而删除旧自驾快照兼容。
- 在没有确认的情况下覆盖未应用候选方案。
- 将自驾偏好无条件展示给所有用户。
- 引入数据库、登录、前端框架或新的必需付费服务。

## 16. 开放问题

以下问题不阻塞本 PRD 评审，但应在实现任务拆分前锁定：

1. 混合交通场景下，“优化自驾路线”是仅处理连续自驾段，还是沿用当前整条路线模型？本 PRD 默认沿用当前能力并明确说明边界。
2. 非自驾用户是否需要“切换为自驾规划”的低权重入口？本 PRD 默认提供文字入口，但不展示禁用主按钮。
3. 存在未应用候选方案时，使用原生确认对话框还是项目内对话框？建议复用项目内对话框，以获得一致的焦点管理和文案。
